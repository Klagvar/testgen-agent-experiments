# План экспериментальной оценки `testgen-agent`

Документ описывает дизайн, протокол и инфраструктуру экспериментальной части ВКР.
Является «source of truth» по эксперименту: список моделей, репозиториев, метрик,
конфигураций и сценариев анализа результатов фиксируется здесь, в коде и
артефактах ничего не дублируется.

Все артефакты эксперимента (этот документ, dataset, скрипты orchestration,
сырые JSON-отчёты, агрегированные таблицы и графики) живут в отдельном
репозитории `testgen-agent-experiments`. Основной репозиторий
`testgen-agent` подключается как submodule на конкретный тег, чтобы версия
агента, которой получены результаты, была однозначно определена.

---

## 1. Цели и научные вопросы

Эксперимент должен дать ответ на четыре исследовательских вопроса (RQ):

- **RQ1. Каждая ли компонента предложенного пайплайна вносит измеримый вклад
  в качество генерируемых тестов?** Проверяется ablation-исследованием:
  поочерёдным отключением одной компоненты при сохранении остальных и
  сравнением метрик с полной конфигурацией.
- **RQ2. Зависит ли вклад компонент от размера и семьи языковой модели?**
  Тот же ablation выполняется на нескольких моделях разного размера и
  происхождения; сравниваются «дельты» от отключения компонент между моделями.
- **RQ3. Способен ли метод вытаскивать качество дешёвых моделей до уровня
  более дорогих, и где находится нижняя граница его применимости?**
  Сопоставляются метрики полной конфигурации на open-source моделях
  разного размера (7B–70B и MoE) и на~closed-source моделях среднего
  ценового сегмента. В~линейку включается floor-reference модель
  ниже~порога устойчивого понимания Go, чтобы~явно ограничить
  операционный диапазон метода (см. §3).
- **RQ4. Насколько метод робастен к доменам реальных Go-проектов?** Один и
  тот же протокол прогоняется на 8 репозиториях с разными доменами (web/HTTP,
  CLI, parsing/encoding, embedded storage, distributed consensus, application
  servers) и размерами (от 2 до 150 KLoC).

## 2. Дизайн эксперимента

Полнофакторный куб `модель × конфигурация × репозиторий × прогон`:

| Фактор          | Уровни                                               | Кол-во |
|-----------------|------------------------------------------------------|--------|
| Модель          | см. §3                                               | 6      |
| Конфигурация    | `full` + 5 ablations (см. §4)                        | 6      |
| Репозиторий     | см. §6, отбираются по фиксированному merged PR       | 8      |
| Прогон          | повтор для статистики (`--runs 3`)                   | 3      |
| **Итого**       |                                                      | **864** |

Один прогон = один вызов `testgen-agent` на один `base..head` пары коммитов
тестируемого репозитория с одной комбинацией модель/конфигурация/seed.

Перед каждым прогоном рабочая копия тестируемого репозитория приводится
к чистому состоянию `git reset --hard <head> && git clean -fdx`. Это
исключает протекание состояния между конфигами (сгенерированные тестовые
файлы, `.testgen-cache.json`, артефакты `merger`-а) и между повторами
одной конфигурации. Сброс выполняется в `internal/benchmark/runner.go`
через callback `ablation.RunOptions.Reset`, см. §10.5.

Каждый прогон генерирует один JSON-отчёт по схеме `internal/report/json.go`,
все 864 отчёта агрегируются `cmd/ablate-report` и `cmd/benchmark-report`.

## 3. Языковые модели

Выбраны **шесть** моделей из **пяти** организаций (у Alibaba — две
модели: общая малая и code-ориентированная средняя). Все доступны через
единый OpenAI-совместимый API на OpenRouter
(`https://openrouter.ai/api/v1`). Это снимает необходимость регистрации
отдельных аккаунтов и нормализует биллинг.

| ID на OpenRouter                            | Семья / лаборатория | Размер           | Тип          | Роль в~эксперименте | Цена in / out (за 1M токенов) |
|---------------------------------------------|---------------------|------------------|--------------|---------------------|-------------------------------|
| `qwen/qwen-2.5-7b-instruct`                 | Qwen / Alibaba      | 7B               | open, generic| **floor reference** | $0.04 / $0.10                 |
| `qwen/qwen3-coder-30b-a3b-instruct`         | Qwen / Alibaba      | 30B (3B active)  | open, code   | candidate           | $0.07 / $0.27                 |
| `meta-llama/llama-3.3-70b-instruct`         | Llama / Meta        | 70B              | open, generic| candidate           | $0.10 / $0.32                 |
| `openai/gpt-4o-mini`                        | GPT / OpenAI        | undisclosed      | closed       | candidate (опорная) | $0.15 / $0.60                 |
| `deepseek/deepseek-chat`                    | DeepSeek            | 671B (37B active)| open, generic| candidate           | $0.32 / $0.89                 |
| `anthropic/claude-3.5-haiku`                | Claude / Anthropic  | undisclosed      | closed       | candidate           | $0.80 / $4.00                 |

**Почему и DeepSeek, и Claude.** Это не взаимозаменяемые «ещё одна
дорогая модель»: DeepSeek — открытая архитектура MoE с китайской
экосистемой; Anthropic — западная проприетарная линейка с иной
post-training. Вместе с OpenAI (GPT-4o-mini), Meta (Llama) и двумя
Qwen они дают заявленное **географическое и методологическое
разнообразие** для RQ2/RQ3.

**Зачем `qwen-2.5-7b-instruct` как floor reference.** Эта модель
сознательно подобрана **ниже порога устойчивого понимания Go**:
предварительные локальные прогоны показали, что~она систематически
порождает тесты, которые либо не~компилируются, либо не~запускаются.
Цель её включения — не~доказать, что~метод работает на~ней (он не
~работает), а~**определить операционный диапазон метода**: до~какого
размера базовой модели прирост от~компонентов пайплайна остаётся
измеримым. Таким образом мы~получаем нижнюю точку кривой
«размер → качество» в~RQ3 и~снимаем очевидный вопрос комиссии
«а~что~будет на~ещё~меньшей модели?». Граница задаётся не
строго размером в~7B параметров — модели той~же размерности,
дотюненные на~коде (например, code-варианты Qwen 7B), могли~бы её
преодолеть; это~направление дальнейшего исследования. В~таблицах
раздела «Анализ результатов» эта~модель проходит в~отдельной колонке
`floor` и~исключается из~агрегатов по~ablation-дельтам, где~значения
почти~нулевые и~любая дельта тонет в~шуме.

**Актуальность ID.** Идентификаторы в OpenRouter периодически меняются;
перед прогоном сверяйте список на
`https://openrouter.ai/docs#models` или через `GET /v1/models`. Ранее
использовавшийся `qwen/qwen-2.5-coder-7b-instruct` в каталоге больше не
валиден — для floor-reference используем `qwen/qwen-2.5-7b-instruct`.

GPT-4o-mini выбран в качестве «опорной» модели для основной части
ablation-таблицы, потому что (а) на ней гонялись оригинальные публикации
по теме (ASTER использует GPT-4 и GPT-4-Turbo, mini — её недорогой
наследник), (б) её ценовой класс позволяет включить её во все факторы куба.

## 4. Конфигурации (ablation)

Полный список конфигов хранится в `internal/ablation/configs.go`. Здесь
дублируется для самодостаточности документа.

| Конфигурация              | Что отключено                                                    | Какую гипотезу проверяет                              |
|---------------------------|------------------------------------------------------------------|-------------------------------------------------------|
| `full`                    | ничего (baseline)                                                | Верхняя граница метрик метода                         |
| `no-types`                | анализ через `go/types`, остаётся синтаксический fallback        | Вклад типизированного контекста в подсказку модели    |
| `no-smart-diff`           | пер-функциональный отбор по git-diff, обрабатывается всё         | Вклад точечного отбора в стоимость и стабильность     |
| `no-structured-feedback`  | парсинг `go test -json`, в repair-prompt уходит сырое stderr     | Вклад структурированной обратной связи в pass-rate    |
| `no-pruning`              | сохраняем все тесты, включая упорно падающие                     | Вклад AST-обрезки в pass-rate и naturalness           |
| `no-coverage`             | итеративный цикл «coverage gap → новый тест»                     | Вклад coverage-loop в покрытие и mutation score       |

6 конфигов = 5 «one-off» + 1 baseline. Это позволяет вычислить **дельту
вклада** каждой компоненты как `metric(full) − metric(no-X)`.

**Почему здесь нет `no-cache`.** Функциональный кэш (`internal/cache`) —
инженерная оптимизация, активирующаяся **только** на повторном прогоне
агента в репозитории, где уже существует `.testgen-cache.json` от
предыдущего запуска. В нашем экспериментальном протоколе перед каждым
прогоном выполняется `git clean -fdx`, который удаляет кэш-файл вместе с
остальными артефактами; следовательно, при первом и единственном прогоне
кэш всегда пуст и `no-cache` тождественен `full`. Включать его в куб
ablation бессмысленно. Эффективность кэша измеряется отдельным
микро-экспериментом, см. §8.5.

## 5. Метрики

Все метрики собираются автоматически и записываются в JSON-отчёт
(`internal/report/json.go`). Здесь — что они означают и как используются.

### 5.1 Покрытие изменённого кода — `diff_coverage_pct`

**Что считаем.** Доля изменённых (в `base..head`) строк продуктового кода,
покрытых тестами, сгенерированными агентом.

**Метод.** `go test -coverprofile`, затем фильтрация covered-строк по списку
изменённых строк из git-diff.

**Что показывает.** Базовая мера полезности генерации. Главный «ось Y»
итоговых таблиц.

**Семантика нулевого значения.** `0.0` = «изменения были, тесты прогнались,
но ни одна изменённая строка не покрыта»; `null` = «coverage не считали»
(`--no-coverage`, либо ошибка сборки тестов до этапа покрытия). Различие
между `0.0` и `null` критично для RQ1: при `no-coverage` мы не должны
учитывать coverage-метрику в среднем.

### 5.2 Mutation score — `mutation_score_pct`

**Что считаем.** Доля выявленных мутантов среди порождённых на изменённых
строках кода. Мутант считается выявленным, если хотя бы один сгенерированный
тест падает на мутантной версии и проходит на исходной.

**Операторы.** Реализованы в `internal/mutation/mutation.go`: инверсия
бинарных отношений, замена логических связок, подмена возвращаемого
значения на нулевое, замена арифметических операторов.

**Что показывает.** Качество ассертов и поведенческой проверки. Coverage
ловит «строка выполняется», mutation ловит «строка проверяется».

**Семантика нулевого значения.** `0.0` = «мутанты порождены, но ни один
тест их не отличает от оригинала»; `null` = «мутации не считали или
ноль мутантов было порождено» (например, если на изменённых строках нет
операторов из набора).

### 5.3 Branch coverage — `branch_coverage_pct`

Доля покрытых веток условных операторов (`if/switch/select`) в изменённых
функциях. Считается дополнительно к строковому покрытию, чтобы поймать
случаи «строка покрыта, но только одна из веток».

### 5.4 Error-path coverage — `error_path_coverage_pct`

Доля покрытых веток обработки ошибок (паттерн `if err != nil`) в
изменённых функциях. Это специфичная для Go метрика, явно вычленяется
из общего branch-coverage, поскольку отражает идиоматическую часть
тестирования Go-кода.

### 5.5 Pass-rate — `tests_validated / tests_generated`

Доля сгенерированных тестов, которые скомпилировались и прошли. Считается
неявно по `tests_validated` и `tests_total` в `JSONTotals`. Главная метрика
«полезности на единицу труда модели».

### 5.6 Tests pruned — `tests_pruned`

Сколько тестов было сгенерировано, но удалено `pruner` после того, как
бюджет ретраев исчерпан. Используется для интерпретации pass-rate (без
этой метрики `pass-rate = 100%` тривиально достижим обнулением выхода).

### 5.7 Token efficiency — `token_efficiency_tokens_per_test`

Суммарные токены (prompt + completion) на один прошедший тест. Совмещает
стоимость и качество в одно число, нужна для сравнений между моделями
с разной плотностью генерации.

### 5.8 Naturalness suite — `naturalness.*`

Семь синтаксических метрик из главы 3 (assertion ratio, no-assertion %,
duplicate %, nil-only %, error-assertion %, test-name score, var-name
score). Отвечают на вопрос «похожи ли сгенерированные тесты на
человеческие». Используются как secondary outcome.

### 5.9 Cost / time — поля `prompt_tokens`, `completion_tokens`, `duration_seconds`

Стоимость прогона и длительность. Используются в сводных таблицах для
аргумента «дёшево и быстро».

## 6. Датасет: тестируемые репозитории

### 6.1 Критерии отбора

**Технические (обязательные).**

- Go ≥ 1.21 (наш `go/types` корректно работает только начиная с 1.20+).
- `go.mod` в корне или в однозначном поддиректории.
- Сборка из коробки: `go build ./...` и `go test ./...` без docker, без
  обращений к интернету, без секретных переменных окружения.
- В CI явно гоняется `go test`.
- Не vendor-only зависимости в проде.

**Содержательные.**

- Есть хотя бы 3 merged PR с 1–4 изменёнными `.go`-файлами в продовом коде
  (без `_test.go`, без `.pb.go` и подобной автогенерации).
- Покрытие не 100% (есть что генерировать) и не 0% (есть с чем сравнить).
- Покрытие предпочтительно по нашему набору доменов (см. ниже).

**Раскладка по доменам.** Стремимся к 8 репозиториям, разделённым по
домену и размеру:

| Слот | Домен                    | Размер     |
|------|--------------------------|------------|
| 1    | HTTP-роутер / middleware | малый      |
| 2    | Утилита / data structure | малый      |
| 3    | CLI / framework          | средний    |
| 4    | Парсер / кодек           | средний    |
| 5    | Web framework            | средний    |
| 6    | Embedded storage         | средне-крупный |
| 7    | Distributed / concurrency| крупный    |
| 8    | Прикладное приложение    | крупный    |

### 6.2 Шорт-лист кандидатов

См. `кандидаты-репо.md` — там по каждому слоту список 2–3 проектов, плюс
3–5 candidate-PR на каждом проекте (с конкретными SHA `base` и `head`).
После клонирования и smoke-теста финальный выбор фиксируется в
`dataset.yaml`.

### 6.3 Финальный `dataset.yaml`

Формат (схема `internal/benchmark/dataset.go`):

```yaml
- name: gorilla-mux
  url: https://github.com/gorilla/mux.git
  base: <sha-base>
  head: <sha-head>
  description: HTTP-роутер; PR <ссылка>
- name: ...
```

Каждый элемент — это «срез» истории тестируемого репо на конкретном PR.
В одном репозитории можно использовать несколько PR как разные элементы
(`name: gorilla-mux-pr-1`, `gorilla-mux-pr-2`), но в основной серии
ограничиваемся одним PR на репо ради чистоты дизайна.

## 7. Протокол прогона

### 7.1 Подготовка

1. Создаётся репозиторий `testgen-agent-experiments` с этой структурой:
   ```
   testgen-agent-experiments/
     README.md
     план.md             # копия этого документа
     dataset.yaml
     кандидаты-репо.md
     scripts/
       smoke.ps1         # smoke-тест на 1 репо × все конфиги × 1 модель × 1 прогон
       run-ablation.ps1  # ablation на одну модель
       run-all.ps1       # полный куб
     workdir/            # клонированные тестируемые репо, в .gitignore
     результаты/
       raw/<model>/<repo>/<config>-runN.json
       агрегированные/<model>.csv
       графики/
     анализ/
       analyse.ipynb     # Jupyter-ноут с графиками для главы 5
   ```

2. Регистрация на OpenRouter, депозит $30. Один API-ключ, кладётся в
   `.env` в `experiments`-репо (в `.gitignore`).

3. `go install` `testgen-agent` из тега, который зафиксирован в submodule.

### 7.2 Smoke-тест

Перед основным прогоном — ограниченный smoke на одной самой дешёвой модели
и одном репозитории:

- модель: `qwen/qwen-2.5-7b-instruct` (самая дешёвая в основном кубе),
- репо: один из малых,
- 6 конфигов × 1 прогон × 1 репо = 6 запусков.

Цель — убедиться, что:

- `cmd/benchmark` корректно клонирует, переключает на `head`, прогоняет
  агент;
- между конфигами реально срабатывает `git reset --hard && git clean -fdx`
  (проверяется по отсутствию `.testgen-cache.json` и `*_generated_test.go`
  в начале каждого следующего прогона);
- агент возвращает валидный JSON для всех 6 конфигов;
- реальное число токенов и время на запуск примерно соответствует оценке
  (моя оценка: ~30K prompt + ~10K completion, ~3–5 минут на запуск).

После smoke-теста уточняется бюджет; если средний расход токенов на запуск
выше оценки в 1.5+ раза, либо урезаем датасет, либо добавляем бюджет.

### 7.3 Полный прогон

Порядок прогонов **от самой дешёвой модели к самой дорогой**:
Qwen-7B (floor) → Qwen3-Coder-30B → Llama-70B → GPT-4o-mini →
DeepSeek-chat → Claude-3.5-Haiku.

Логика: если первая же модель сломает сборку или агент — узнаем за ~$0.30,
а не за ~$9. Ошибка в скрипте orchestration на ранних моделях обходится
дёшево. Дополнительный плюс: Qwen-7B как~floor-reference (см. §3) запускается
первой, и~её 144 прогона быстро подтверждают, что~инфраструктура работает
end-to-end даже для~модели с~ожидаемо высоким failure rate. **Claude
оставлен последним:** она дороже всех и даёт ~60% суммарной стоимости
куба при верхней оценке токенов — после smoke можно уточнить бюджет.

Для каждой модели:

```powershell
testgen-bench `
  --dataset dataset.yaml `
  --output-dir результаты/raw/<model>/ `
  --runs 3 `
  --seed-base 42 `
  --temperature 0
# модель и API передаются в TESTGEN_MODEL / TESTGEN_API_URL
```

Результат: 8 репо × 6 конфигов × 3 прогона = 144 JSON-файла на модель.

### 7.4 Обработка отказов

Возможные сбои и реакция:

- **API rate limit** — экспоненциальный backoff в OpenAI-клиенте уже есть;
  если упёрлись в дневной лимит — пауза до сброса.
- **Ошибка сборки тестируемого репо** на конкретном `head` — фиксируем в
  логах, исключаем репо/PR из датасета на следующем перепрогоне, не
  заполняем «нулями» (это бы исказило среднее).
- **Зависший прогон агента** — таймаут запуска `--timeout-seconds`,
  по умолчанию 600. На больших репо может потребоваться увеличить.
- **OpenRouter недоступен** — пауза, ретрай с того же `--run-index`.

## 8. Анализ результатов

### 8.1 Базовая агрегация

`cmd/ablate-report` собирает все JSON-отчёты в одну CSV-таблицу
`(repo, model, config, run_index, метрика_1, ..., метрика_N)`.

Из неё строятся четыре основные таблицы и три графика для главы 5 ВКР.

### 8.2 Таблицы

- **T1. Ablation на главной модели (GPT-4o-mini).** Среднее по репо и
  прогонам, столбцы — конфигурация, строки — метрика. Δ относительно `full`.
- **T2. Кросс-модельный ablation.** Сводная таблица «модель × конфигурация»
  для одной главной метрики (diff coverage). Тепловая карта.
- **T3. Per-repo робастность.** Сводная «модель × репо» для главной
  метрики на конфигурации `full`. Тепловая карта.
- **T4. Cost-effectiveness.** «Модель × средняя метрика, средние токены,
  средняя стоимость, время». Главное: показать соотношение цена/качество.

**Floor reference в~таблицах.** Qwen 2.5 7B (см. §3) не~входит в~список
candidate-моделей и~обрабатывается отдельно:

- В~T1 не~участвует (главная ablation — на~candidate-моделях).
- В~T2 присутствует отдельной строкой `floor` с~явной пометкой; ablation-дельты
  для~неё ожидаемо нулевые либо отрицательные внутри~шума, что~само по~себе
  является результатом и~обсуждается в~разделе RQ3.
- В~T3 включается, чтобы~показать что~отказ происходит на~всех 8 репо
  (а не~специфичен к~одному).
- В~T4 находится в~отдельной нижней строке `(floor)` — она~тратит токены,
  но~не~производит валидных тестов, и~поэтому не~должна попадать в~среднюю
  cost-effectiveness candidate'ов.

### 8.3 Графики

- **G1. Bar chart «вклад каждой компоненты».** По T1: `metric(full) − metric(no-X)`,
  bar per X. С error-bars по 3 прогонам. Строится только по~candidate-моделям.
- **G2. Scatter «cost vs quality» по моделям.** Каждая модель — точка
  (стоимость на прогон, средний diff coverage). Floor-модель показывается
  отдельным маркером (треугольник вместо круга) в~зоне «низкая
  стоимость, ~0% качество», чтобы~явно обозначить нижнюю границу
  операционного диапазона.
- **G3. Линия «scaling study».** X — модели в порядке возрастания цены,
  Y — главная метрика на конфигурации `full`. С error-bars. Floor-модель
  включается как~самая левая точка кривой, демонстрируя резкий перегиб
  ниже~7B-уровня языковой компетенции.

### 8.4 Статистическая обработка

Три прогона на ячейку — слишком мало для строгого hypothesis testing.
Используем simple confidence intervals на t-распределении (n=3) или
bootstrap-quantiles при агрегации по репо. В тексте ВКР не претендуем
на статистическую значимость каждой Δ, но указываем спред.

Для~floor-reference статистика по~ablation-дельтам не~приводится: при
нулевом или почти-нулевом pass-rate `metric(full) − metric(no-X)`
вырождается в~разность двух нулей, что~аналитически бессмысленно. В
~отчёте об~этом~даётся одна строка-сноска.

### 8.5 Микро-эксперимент: эффективность кэша

Цель — измерить экономию токенов при использовании функционального кэша
(`internal/cache`) на типичном CI-цикле, когда один и тот же PR
переcобирается несколько раз.

Дизайн (одна модель — `gpt-4o-mini`, один репозиторий — `gorilla/mux`,
один зафиксированный `head`):

1. **Прогон A.** `full` без кэша (`.testgen-cache.json` отсутствует).
   Это нормальный «холодный» запуск, фиксируем `prompt_tokens`,
   `completion_tokens`, `duration_seconds`.
2. **Прогон B.** `full` с кэшем, накопленным в Прогоне A, на том же
   `head` (без изменений в коде). Ожидаем близкий к нулю расход
   LLM-токенов и существенное сокращение времени.
3. **Прогон C.** `full` с тем же кэшем, но с `head`, сдвинутым на один
   следующий коммит. Часть функций изменилась, кэш частично
   инвалидируется. Фиксируем долю сэкономленных токенов и время.

Результат: одна таблица из трёх строк в подразделе «Эффективность кэша»
главы 5 ВКР. Стоимость — 3 запуска ≈ $0.05.

Этот микро-эксперимент **не входит** в основной куб ablation: он
концептуально другой (измеряет инкрементное поведение между прогонами,
а не одну итерацию), и включение `no-cache` в куб с `git clean` между
запусками было бы тождественно `full`.

## 10. Технические решения

### 10.1 Почему отдельный репо для эксперимента

Альтернативы:

- *Всё в `testgen-agent`* — основной репо распухнет на 50–100 МБ raw-JSON,
  путаются коммиты «код агента» и «коммиты по результатам».
- *По репо на каждый тестируемый проект* — нет смысла, мы их даже не
  меняем.
- *Отдельный `testgen-agent-experiments`* — выбран. Чистый основной
  репозиторий, отдельный README про эксперимент, привязка к версии
  агента через submodule, в ВКР указывается единая ссылка.

### 10.2 Воспроизводимость

- `temperature=0`, `seed=42 + run_index − 1` (то есть 42, 43, 44 для
  трёх прогонов). Это уже поддержано CLI-флагами `--temperature`, `--seed`,
  `--run-index` и инфраструктурой `--runs`/`--seed-base` в
  `cmd/ablate` и `cmd/benchmark`.
- Каждый JSON-отчёт записывает `Temperature`, `Seed`, `RunIndex`,
  `AblationConfig`, `Model`, `Provider` в раздел `config`.
- Зафиксированы коммиты и тестируемого репо (`base`/`head`), и
  агента (тег submodule), и моделей (точные ID OpenRouter), и
  upstream-провайдеров OpenRouter, через которых обслуживался каждый
  запрос (см. §10.7).

### 10.3 Модель-агностичность

`internal/llm/client.go` — обычный HTTP-клиент к OpenAI-совместимому API.
OpenRouter полностью совместим. Никаких vendor-specific SDK.

### 10.4 Бюджет токенов

`internal/llm/client.go` уже логирует prompt/completion токены каждого
запроса. Скрипт orchestration (`run-all.ps1`) суммирует токены по
завершении прогона на модель и сравнивает с бюджетом; если превышено —
останавливается.

### 10.5 Сброс рабочей копии между прогонами

Чтобы независимые прогоны (разные ablation-конфиги или повторы одной
конфигурации) не загрязняли друг друга, перед каждым запуском агента
выполняется:

```bash
git -C <clone-dir> reset --hard <head-SHA>
git -C <clone-dir> clean -fdx
```

`reset --hard` отменяет любые правки, которые `merger` мог внести в
существующие файлы. `clean -fdx` удаляет все неотслеживаемые файлы и
директории, в т.ч. `.testgen-cache.json` и сгенерированные тестовые
файлы (`*_generated_test.go`).

Реализация: `Cloner.Reset(dir, headSHA)` в
`internal/benchmark/clone.go`, передаётся в `ablation.RunOptions.Reset`
как callback, который `ablation.runOnce` вызывает перед запуском
агента. Покрыто юнит-тестами `TestCloner_Reset`,
`TestRunner_ResetCallback`, `TestRunner_ResetFailureAborts`.

Без этого сброса:

- `no-types`, `no-pruning` и другие ablation-конфиги получали бы
  кэш-хиты от ранее прогнанной `full`-конфигурации и метрики
  выглядели бы идентично базовой;
- повторы внутри `--runs N` отличались бы только по `seed`, но не по
  состоянию репо, и variance был бы существенно занижен.

### 10.6 Бюджет времени на~застрявшие тесты

LLM иногда генерирует тест, который зацикливается на~ожидании события
(time.Sleep в~цикле, недостижимая монотонная инварианта, dead-lock на
~канале). Validator вызывает `go test` с~общим таймаутом, и~без~явного
ограничения один такой тест внутри одного прогона тратит весь
дефолтный бюджет 300 с, а~затем уходит в~retry-loop ещё~3 раза.

Smoke-прогон 2026-05-09 показал реальный случай: Qwen-2.5-7B
сгенерировала тест на~UUIDv7, который завис на~монотонии,
и~один-единственный прогон (`no-coverage` на~`google-uuid`) занял
949 с против ~50 с у~остальных конфигов.

В~основном эксперименте таймаут go-test зажимается до~60 с через
CLI-флаг `--test-timeout`, который агент обрабатывает с~версии
[`7ffe29e`](https://github.com/Klagvar/testgen-agent/commit/7ffe29e)
и~`cmd/benchmark` пробрасывает через
`--extra "--test-timeout 60 --temperature 0 --seed-base 42"`. Это
ограничивает наихудший прогон ≈ 3 мин (60 с × 3 retry) и~гарантирует
что~суммарное время куба не~раздуется на~застрявших тестах.

### 10.7 Закрепление upstream-провайдера OpenRouter

OpenRouter — это~агрегатор, который для~каждой модели маршрутизирует
запросы между несколькими физическими провайдерами (например, для
`qwen/qwen-2.5-7b-instruct` доступны Phala, AtlasCloud, Together).
Разные провайдеры используют разные inference-стеки (vLLM / TGI /
SGLang), разные kvантования (FP8 / FP16 / bfloat16) и~разное GPU-
оборудование. Из-за этого даже при `temperature=0` и~фиксированном
`seed` запрос, ушедший на~Phala, и~запрос, ушедший на~Together,
могут вернуть разные токены. Auto-routing OpenRouter — это~скрытый
источник вариативности, который делает наблюдаемые различия в
~метриках смесью эффекта модели и~эффекта бэкенда.

Чтобы устранить эту переменную, агент с~версии
[`4b21392`](https://github.com/Klagvar/testgen-agent/commit/4b21392)
поддерживает CLI-флаг `--provider`, который сериализуется в~теле
HTTP-запроса как~OpenRouter-`provider.only`:

```
--provider Phala                              # строгий пин
--provider "Phala,DeepInfra" \
  --provider-allow-fallbacks                  # пул из~двух с~fallback
```

По умолчанию `allow_fallbacks=false` — то~есть если выбранный
провайдер недоступен, OpenRouter возвращает ошибку, а~не~подменяет
бэкенд. Это~предотвращает «тихое» переключение в~середине прогона.

В~`scripts/run-model.sh` это~прокидывается через переменную окружения
`PROVIDER`, например:

```bash
PROVIDER=Phala bash scripts/run-model.sh qwen/qwen-2.5-7b-instruct
```

Логирование. Заполненный список `Provider` сохраняется в~поле
`config.provider` каждого JSON-отчёта, чтобы~постфактум-агрегатор мог
группировать прогоны по~бэкенду и~атрибутировать различия в~метриках
либо самой модели, либо инференс-стеку.

Выбор провайдера для~каждой модели:

| Модель                              | Провайдеры на~OpenRouter | Используемый | Стоимость (\$/M token in/out) |
|-------------------------------------|--------------------------|--------------|--------------------------------|
| `qwen/qwen-2.5-7b-instruct`         | Phala, AtlasCloud, Together | **Phala** | 0.04 / 0.10 |
| `qwen/qwen3-coder-30b-a3b-instruct` | Novita, SiliconFlow, Alibaba, Bedrock | **Novita** | 0.07 / 0.27 |
| `meta-llama/llama-3.3-70b-instruct` | DeepInfra, AkashML, Novita, Parasail, Cloudflare, SambaNova, Groq, Friendli, Google, Together (12 RU-friendly из 15) | **DeepInfra** | 0.10 / 0.32 |
| `deepseek/deepseek-chat`            | TBD                      | TBD          | TBD                            |
| `openai/gpt-4o-mini`                | OpenAI (single)          | OpenAI       | прайс OpenAI                   |
| `anthropic/claude-3.5-haiku`        | Anthropic (single)       | Anthropic    | прайс Anthropic                |

Для~моделей с~единственным провайдером (`gpt-4o-mini`,
`claude-3.5-haiku`) `--provider` не~задаётся — это~no-op. Для
остальных провайдер выбирается перед запуском по~двум критериям:
(1) самый дешёвый среди работающих, (2) подтверждённая совместимость
с~RU-IP при выключенном VPN (см. §10.8).

### 10.8 Сетевая изоляция

Запросы выполняются из~WSL2 / Ubuntu 24.04 на~хосте Windows 11.
Хост использует Clash Verge в~режиме TUN, который перехватывает весь
IP-трафик на~уровне виртуального адаптера и~маршрутизирует его через
финский exit-нод (AS213520 Senko Digital LLC, Helsinki). Это~даёт:

- стабильный non-RU IP, что~важно для~провайдеров OpenAI / Anthropic,
  блокирующих RU;
- единый исходящий IP на~всё время эксперимента, что~устраняет
  гипотезу «разные провайдеры дают разный rate-limit на~разные IP»;
- независимость от~переключений мобильного оператора и~прочих
  сетевых событий хоста.

Для~Qwen 2.5 7B через Phala провайдер не~блокирует RU-IP, поэтому
эту модель технически можно прогнать и~без~VPN. Однако для
консистентности всех 6 моделей и~устранения любых artefacts от
gateway-уровня (CDN, geo-DNS) все прогоны выполняются с~одного и
того~же исходящего IP с~включенным TUN-прокси.
