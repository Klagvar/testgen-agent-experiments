## §5.1. Постановка эксперимента (~4-5 страниц)

### §5.1.1. Исследовательские вопросы

- **RQ1**: Какова зависимость качества автоматически сгенерированных тестов от выбора языковой модели?
- **RQ2**: Каков вклад каждой из шести компонент пайплайна в итоговое качество?
- **RQ3**: Способен ли агент-харнесс компенсировать слабость более простых LLM, приводя их к качеству, сопоставимому с frontier-моделями?
- **RQ4**: Каковы характеристики качества тестов помимо успешности компиляции — покрытие изменённого кода, натуральность, мутационная стойкость?
- **RQ5**: Насколько устойчивы результаты между независимыми запусками с разными зернами генератора псевдослучайных чисел?
- **RQ6**: Какова экономическая эффективность функционального кэширования в типичном сценарии непрерывной интеграции?

### §5.1.5. Параметры воспроизводимости

- Температура `0` (детерминированный sampling в пределах допустимого для LLM)
- Зерно ГПСЧ `42 + run_index` (1, 2, 3) для разных запусков одной (модель, репо, конфигурация)
- Жёсткое закрепление upstream-провайдера через OpenRouter (`provider.only`, без fallback)
- `git reset --hard <head> && git clean -fdx` между запусками (исключает накопление состояния)
- WSL2/Ubuntu в качестве среды исполнения
- **Лимит ответа модели:** в клиенте установлен достаточно высокий потолок генерации (16384 токенов) — это исключает обрезку длинных табличных тестов на нативных API-провайдерах, которые соблюдают этот лимит строго (одна фраза, без отдельного раздела про validity).

### §5.1.6. Метрики

(Краткое перечисление со ссылкой на §3.6, без повторного определения.)
- **Доля успешных запусков** (success rate) — главная агрегированная метрика
- **Число валидных тестов** — количество тестов, прошедших компиляцию и выполнение
- **Покрытие изменённого кода** (diff coverage)
- **Покрытие ветвлений** (branch coverage)
- **Покрытие путей обработки ошибок** (error path coverage)
- **Мутационная стойкость** (mutation score)
- **Натуральность** (5 подметрик)
- **Эффективность по токенам** (токенов на один валидный тест)
- **Время работы агента** (wall-clock)

### §5.1.7. Самотестирование инструмента

**Один абзац:**

Программная реализация SAGA сопровождается собственным набором юнит-тестов. На момент проведения эксперимента 36 файлов автотестов охватывают 23 внутренних пакета; среднее покрытие по операторам составляет 72,1 %. Критичные модули (модуль удаления невалидных тестов, модуль кэширования, модуль покрытия ветвлений, модуль разбора структурированного вывода тестового фреймворка) имеют покрытие выше 90 %. Это служит косвенным свидетельством корректности тех компонент, чей вклад количественно оценивается далее в этой главе.

### §5.1.8. Воспроизводимость

Все артефакты эксперимента — описания репозиториев, скрипты запуска, ~1 010 машиночитаемых отчётов, агрегационные скрипты — опубликованы в открытом репозитории~\cite{saga-experiments}. Эксперимент полностью воспроизводим командами, документированными в README указанного репозитория.

## §5.4. Эффект функционального уравнивания моделей среднего тира (~4-5 страниц)

### §5.4.1. Постановка вопроса

Способен ли наш агент-харнесс компенсировать слабость более простых LLM? Без полноценного bare-LLM baseline (отдельный эксперимент «модель пишет тест без агента») мы не можем ответить строго, но через исследование вклада компонент есть полезный proxy: конфигурация `no-pruning` представляет собой минимально-функциональный harness без AST-фильтрации, то есть ближе всего к «голому LLM с обвязкой компиляции».

### §5.4.3. Три формулировки гипотезы

### §5.4.6. Практическая рекомендация

> Для CI-сценария выбор LLM в среднем тире определяется уже стоимостью и латентностью; различия в capability нивелируются агентом. Это снижает чувствительность пайплайна к выбору LLM-провайдера и ценовой/доступностной волатильности.

### §5.4.7. Caveat

Конфигурация `no-pruning` — proxy, не bare-LLM. Истинный baseline — future work.

- кросс-провайдерная совместимость (любой OpenAI-compatible эндпоинт)
- собственная инфраструктура воспроизводимого многомодельного бенчмарка

## Выводы по главе 5 (~1,5-2 страницы)

Структурированное перечисление **6 центральных результатов** + **3 практические рекомендации**.

Результаты:
1. Эмпирическое расслоение 7 LLM (6 вендоров) от 15 до 92 % успешности — обеспечивает scaling-анализ
2. Универсальная критичность модуля удаления невалидных тестов (pruner): -40 до -71 п.п. при отключении на всех моделях
3. Не-монотонное поведение компонент: зависимость утилитарности от capability модели
4. Эффект функционального уравнивания моделей среднего тира в production-ready категорию
5. Эффективность кэширования: -52 % wall-clock в типичном CI-сценарии
6. Сравнение с альтернативами подтверждает уникальность подхода (diff-aware + типизированный анализ + мутации на изменениях)

Рекомендации:
- Pruner — единственный неотключаемый компонент при production-применении
- Выбор LLM в среднем тире определяется ценой/латентностью, не capability
- В CI-сценариях immutable-branches кэширование даёт значимую экономию
