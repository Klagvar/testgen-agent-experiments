## §5.1. Постановка эксперимента (~5 страниц)

### §5.1.1. Исследовательские вопросы

- **RQ1**: Какова зависимость качества автоматически сгенерированных тестов от выбора языковой модели?
- **RQ2**: Каков вклад каждой из шести компонент пайплайна в итоговое качество?
- **RQ3**: Способен ли агент-харнесс компенсировать слабость более простых LLM, приводя их к качеству, сопоставимому с frontier-моделями?
- **RQ4**: Каковы характеристики качества тестов помимо успешности компиляции — покрытие изменённого кода, ветвлений, путей обработки ошибок, мутационная стойкость, натуральность?
- **RQ5**: Насколько устойчивы результаты между независимыми запусками с разными зернами генератора псевдослучайных чисел?
- **RQ6**: Какова экономическая эффективность функционального кэширования в типичном сценарии непрерывной интеграции?

### §5.1.5. Параметры воспроизводимости

- Температура `0` (детерминированный sampling в пределах допустимого для LLM)
- Зерно ГПСЧ `42 + run_index` (1, 2, 3) для трёх запусков одной (модель, репо, конфигурация)
- Жёсткое закрепление upstream-провайдера через OpenRouter (`provider.only`, без fallback)
- `git reset --hard <head> && git clean -fdx` между запусками (исключает накопление состояния)
- WSL2/Ubuntu 24.04, Go 1.24+ в качестве среды исполнения
- Лимит ответа модели — 16384 токенов (одна фраза): нативные API-провайдеры некоторых моделей жёстко обрезают вывод по `max_tokens`, поэтому слишком низкое значение приводит к артефактам валидации; 16K — потолок, при котором ни одна модель в наших данных не упёрлась в обрезку.

### §5.1.6. Метрики

(Краткое перечисление со ссылкой на §3.6, без повторного определения.)

| Метрика | Что измеряет | Шкала |
|---|---|---|
| Доля успешных запусков (run_success) | хоть один валидный тест в запуске | 0–100 % |
| Файловая успешность (file_macro) | доля изменённых файлов, покрытых тестами | 0–100 % |
| Число валидных тестов | прошедших компиляцию и `go test` | целое |
| Покрытие изменённого кода (diff coverage) | доля изменённых строк, исполняемых тестами | 0–100 % |
| Покрытие ветвлений (branch coverage) | доля ветвлений, покрытых тестами | 0–100 % |
| Покрытие путей обработки ошибок | Go-специфика, по error-return-парам | 0–100 % |
| Мутационная стойкость (mutation score) | killed/total на изменённом коде | 0–100 % |
| Натуральность (5 подметрик) | стиль и читаемость | смешанные |
| Эффективность по токенам | токенов / валидный тест | целое |
| Стоимость одного валидного теста | $/test по тарифу провайдера | USD |
| Время работы агента | wall-clock | часы |
| Стандартное отклонение между запусками | дисперсия результата при тех же параметрах | п.п. |

Главный сводный показатель — **доля успешных запусков** (run_success); вспомогательный — **файловая успешность** (file_macro). Различие между ними проговаривается в §5.2 на одном абзаце.

### §5.1.7. Самотестирование инструмента

**Один абзац.** Программная реализация SAGA сопровождается собственным набором юнит-тестов. На момент проведения эксперимента 36 файлов автотестов охватывают 23 внутренних пакета; среднее покрытие по операторам составляет 72,1 %. Критичные модули (модуль удаления невалидных тестов, модуль кэширования, модуль покрытия ветвлений, модуль разбора структурированного вывода тестового фреймворка) имеют покрытие выше 90 %. Это служит косвенным свидетельством корректности тех компонент, чей вклад количественно оценивается далее в этой главе.

### §5.3.3. Универсальная критичность модуля удаления невалидных тестов

- `no-pruning` — единственная компонента с **отрицательным Δ для всех 7 моделей**
- Среднее падение 22,0 п.п., максимум 31,2 п.п. (Claude-3.5)
- Самый крупный и самый стабильный эффект во всём кубе
- **Объяснение через AST-инвариант**: pruner отбраковывает тесты с нарушениями инвариантов (тип, импорт, сигнатура) до `go test`. Без него один невалидный тест ломает компиляцию пакета и аннулирует уже работавшие тесты других файлов внутри запуска.

**Усиление через анализ типов отказов (бонус из 14-failure-modes).** В наших данных категории «компилируется-но-падает» (`all-failed`) и «частично проходит» (`partial`) **отсутствуют во всех 1008 запусках**. Все ошибки концентрируются в категории «нет валидных тестов». Это значит:

- Pruner работает функционально корректно: не пропускает тесты, которые упали бы на runtime
- Главная причина «не работает» — отсутствие сгенерированных тестов вовсе, а не плохие assertions
- Соответственно, отключение pruner превращает тонкую AST-проверку в грубую «компиляция/некомпиляция» границу, и слабые модели стабильно проваливаются

Этот факт даёт сильный аргумент в пользу заявленной новизны pruner-модуля (Глава 3.6.3).

### §5.3.4. Не-монотонное поведение остальных компонент

`no-types`, `no-smart-diff`, `no-structured-feedback` — умеренное среднее влияние (от -1,8 до -5,7 п.п.) с **разнознаковым разбросом** между моделями:

- `no-structured-feedback` ухудшает Llama (-12,5 п.п.) и Qwen-7B (-4,2), но улучшает DeepSeek (+6,2) и Gemini (+2,1)
- `no-types` критически важен для DeepSeek (-16,7 п.п.) и Qwen3-30B (-10,4), но почти не влияет на GPT-4o-mini (+2,1) и Llama (-2,1)

**Объяснение.** Утилитарность дополнительного контекста (типы, структурированный фидбек) зависит от того, **умеет ли модель этим контекстом пользоваться**. Передовая Gemini в среднем не нуждается в дополнительном фидбеке (пишет валидный тест с первой попытки); средний тир выигрывает; floor-модель не имеет capability чтобы воспользоваться даже хорошим контекстом. Это — **обратная корреляция полезности компонент с capability модели** — научно-значимое наблюдение.

### §5.3.5. Цикл расширения покрытия — trade-off

`no-coverage` показывает положительный Δ на всех 7 моделях (среднее +12,8 п.п., максимум +29,2 на Llama). Но это **не означает что компонент бесполезен**:

- Cover-loop оптимизирует **branch coverage**, а не file_macro
- В §5.5.2 (branch coverage) показано: `full` действительно превосходит `no-coverage` по покрытию ветвлений
- В §5.5.7 (variance) показано: cover-loop **стабилизирует** результат — `no-coverage` имеет σ = 18,4 п.п. (max), `full` — 13,3 п.п.
- Это типичный trade-off «больше покрытия + предсказуемость ↔ ниже file-success» — оправданный для CI-сценария

## §5.4. Эффект функционального уравнивания моделей среднего тира (~4 страницы)

### §5.4.1. Постановка вопроса

Способен ли наш агент-харнесс компенсировать слабость более простых LLM? «Чистый» bare-LLM baseline (модель без обвязки) не входит в основной куб — это будущая работа. Однако `no-pruning` представляет собой **минимально-функциональный pipeline** (только LLM + `go test`, без AST-фильтрации) и используется как **proxy-baseline**.

### §5.4.3. Три формулировки гипотезы

1. **«Полное закрытие гэпа» — НЕТ.** Спред Gemini–Qwen-7B расширяется с 62,5 п.п. (no-pruning) до 75 п.п. (full). Harness не сужает разрыв, а поднимает всех (у frontier есть запас, у floor — нет).
2. **«Слабые выигрывают абсолютно больше» — НЕТ.** Самые большие приросты у Claude (+62,5 п.п.) и mid-tier GPT-4o-mini, Qwen3-Coder-30B (+50). Floor-модель Qwen-7B получает +25 п.п. (harness не может компенсировать отсутствие способности писать синтаксически валидный Go-код).
3. **«Уравнивание моделей среднего тира в production-ready категорию» — ДА.**

### §5.4.4. Переход категорий

Категории: **floor** < 25 %, **mid** 25–75 %, **prod-ready** 75–95 %, **frontier** ≥ 95 %.

**Пять из семи моделей** меняют категорию благодаря harness. Особенно показательны Claude (floor → prod-ready, через всю промежуточную) и Gemini (mid → frontier).

### §5.4.6. Diminishing returns at the ceiling

Кратное улучшение (full / no-pruning) убывает с ростом исходной capability: Claude 6×, Qwen3-30B 5×, GPT-4o-mini 3×, Llama 2,8×, DeepSeek 2,3×, Gemini 1,6×. Frontier-модель уже близка к functional ceiling — дополнительный прирост наименьший среди всех.

### §5.4.7. Практическая рекомендация

> Для CI-сценария выбор LLM в среднем тире определяется уже стоимостью и латентностью; различия в capability существенно нивелируются агентом. Это снижает чувствительность пайплайна к выбору LLM-провайдера и к ценовой/доступностной волатильности.

### §5.4.8. Caveat (1 абзац)

Конфигурация `no-pruning` — **proxy-baseline, не bare-LLM**. У него всё ещё есть retry-цикл, structured feedback и типы. Истинный baseline (model-only) — отложен в будущую работу. Несмотря на этот caveat, наблюдаемое улучшение настолько систематическое и крупное, что общее заключение **«harness существенно компенсирует слабость»** остаётся обоснованным.

- **ChatTester** (Yuan, 2024) — Python
- **Qodo Cover** (commercial) — multi-language, не diff-aware
- **CodaMOSA** (Lemieux, 2023) — Python

**Классические инструменты:**
- **EvoSuite** — Java, эволюционный подход
- **Pynguin** — Python, mutation 28–42 %
- **Diffblue Cover** — Java, символьное выполнение

### §5.7.2. Таблица T11 — SAGA vs альтернативы (4-5 строк, 6-7 столбцов)

| Признак | ASTER | TestPilot | EvoSuite | Diffblue | **SAGA (наша)** |
|---|---|---|---|---|---|
| Целевой язык | Java | JS | Java | Java | **Go** |
| Подход | LLM | LLM | эволюционный | символьный | **LLM + типы + AST** |
| Diff-aware | нет | нет | нет | нет | **да** |
| Типизированный анализ | нет | нет | реflection | да | **да (`go/types`)** |
| Mutation на изменениях | нет | нет | нет | нет | **да** |
| Воспроизводимый бенчмарк | нет | нет | стандартный (SF110) | нет | **да (~1010 JSON)** |
| Cross-vendor LLM | нет | нет | — | — | **да (любой OpenAI-compatible API)** |

### §5.7.3. Уникальные характеристики SAGA

- **Diff-aware** — единственная LLM-based система с анализом diff PR на уровне функций
- **Типизированный анализ через `go/types`** — не Tree-sitter (более точная семантика, поддержка интерфейсов)
- **Мутационное тестирование на изменённом коде** — оригинальный вклад (Глава 3.6.5)
- **Cross-vendor совместимость** — любой OpenAI-compatible эндпоинт (демонстрировано на 6 разных вендорах)
- **Воспроизводимая инфраструктура многомодельного бенчмарка** — публикуется отдельно как `\cite{saga-experiments}`

### §5.7.4. Замечание о численных значениях метрик

Прямое сравнение mutation score / coverage с ASTER/TestPilot методологически некорректно:
- У ASTER scope = весь модуль; у нас — только изменённые строки PR
- У TestPilot нет cover-loop (одна попытка); у нас loop до coverage ≥ 80 %
- У EvoSuite/Diffblue вообще нет LLM (другая природа эффективности)

Поэтому сравниваются **подход и сценарий применения**, а не абсолютные числа метрик.

### Результаты

1. **Эмпирическое расслоение 7 LLM (6 вендоров)** от 15 % до 92 % run_success — обеспечивает корректный scaling-анализ
2. **Универсальная критичность модуля удаления невалидных тестов**: -22,0 п.п. mean, -31,2 п.п. max при отключении на всех 7 моделях
3. **Бинарность отказов**: в 1008 запусках нет ни одного случая `partial`/`all-failed` — pruner функционально корректен
4. **Не-монотонное поведение остальных компонент**: полезность вспомогательного контекста обратно коррелирует с capability модели
5. **Эффект функционального уравнивания**: 5 из 7 моделей переходят в более высокую категорию благодаря harness; средний тир выводится в production-ready
6. **Cover-loop как двойной модуль**: оптимизирует branch coverage И стабилизирует результат (σ снижается с 18,4 до 13,3 п.п.)
7. **Мутационная стойкость 89,6–98,2 %** в мини-эксперименте, лидер Gemini-3-Flash (98,2 % на 7/8 репо)
8. **Pareto-frontier стоимость–качество**: 3 недоминируемые модели — Qwen-7B, GPT-4o-mini, Gemini

### Практические рекомендации

1. **Pruner не отключать никогда** — единственный неотключаемый компонент при production-применении
2. **Выбор LLM в среднем тире** определяется ценой и стабильностью, не capability:
   - GPT-4o-mini — Pareto-оптимум для CI ($0,007/тест)
   - Claude — для воспроизводимых сценариев (детерминированность)
   - Gemini — когда требуется максимальное качество
3. **В CI-сценариях с immutable branches** функциональное кэширование даёт −52 % wall-clock без потери качества
