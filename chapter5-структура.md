## §5.1. Постановка эксперимента (~5 страниц)

### §5.1.1. Исследовательские вопросы

- **RQ1**: Какова зависимость качества автоматически сгенерированных тестов от выбора языковой модели?
- **RQ2**: Каков вклад каждой из шести компонент пайплайна в итоговое качество?
- **RQ3**: Способен ли агент-харнесс компенсировать слабость более простых LLM, приводя их к качеству, сопоставимому с frontier-моделями?
- **RQ4**: Каковы характеристики качества тестов помимо успешности компиляции — покрытие изменённого кода, ветвлений, путей обработки ошибок, мутационная стойкость, натуральность?
- **RQ5**: Насколько устойчивы результаты между независимыми запусками с разными зернами генератора псевдослучайных чисел?
- **RQ6**: Какова экономическая эффективность функционального кэширования в типичном сценарии непрерывной интеграции?

### §5.1.5. Параметры воспроизводимости

- Температура `0` (детерминированный sampling в пределах допустимого для LLM)
- Зерно ГПСЧ `42 + run_index` (1, 2, 3) для трёх запусков одной (модель, репо, конфигурация)
- Жёсткое закрепление upstream-провайдера через OpenRouter (`provider.only`, без fallback)
- `git reset --hard <head> && git clean -fdx` между запусками (исключает накопление состояния)
- WSL2/Ubuntu 24.04, Go 1.24+ в качестве среды исполнения
- Лимит ответа модели — 16384 токенов (одна фраза): нативные API-провайдеры некоторых моделей жёстко обрезают вывод по `max_tokens`, поэтому слишком низкое значение приводит к артефактам валидации; 16K — потолок, при котором ни одна модель в наших данных не упёрлась в обрезку.

### §5.1.6. Метрики

(Краткое перечисление со ссылкой на §3.6, без повторного определения.)

| Метрика | Что измеряет | Шкала |
|---|---|---|
| Доля успешных запусков (run_success) | хоть один валидный тест в запуске | 0–100 % |
| Файловая успешность (file_macro) | доля изменённых файлов, покрытых тестами | 0–100 % |
| Число валидных тестов | прошедших компиляцию и `go test` | целое |
| Покрытие изменённого кода (diff coverage) | доля изменённых строк, исполняемых тестами | 0–100 % |
| Покрытие ветвлений (branch coverage) | доля ветвлений, покрытых тестами | 0–100 % |
| Покрытие путей обработки ошибок | Go-специфика, по error-return-парам | 0–100 % |
| Мутационная стойкость (mutation score) | killed/total на изменённом коде | 0–100 % |
| Натуральность (5 подметрик) | стиль и читаемость | смешанные |
| Эффективность по токенам | токенов / валидный тест | целое |
| Стоимость одного валидного теста | $/test по тарифу провайдера | USD |
| Время работы агента | wall-clock | часы |
| Стандартное отклонение между запусками | дисперсия результата при тех же параметрах | п.п. |

Главный сводный показатель — **доля успешных запусков** (run_success); вспомогательный — **файловая успешность** (file_macro). Различие между ними проговаривается в §5.2 на одном абзаце.

### §5.1.7. Самотестирование инструмента

**Один абзац.** Программная реализация SAGA сопровождается собственным набором юнит-тестов. На момент проведения эксперимента 36 файлов автотестов охватывают 23 внутренних пакета; среднее покрытие по операторам составляет 72,1 %. Критичные модули (модуль удаления невалидных тестов, модуль кэширования, модуль покрытия ветвлений, модуль разбора структурированного вывода тестового фреймворка) имеют покрытие выше 90 %. Это служит косвенным свидетельством корректности тех компонент, чей вклад количественно оценивается далее в этой главе.

### §5.3.2. Универсальная критичность модуля удаления невалидных тестов

Самый объёмный подраздел в §5.3 (~1,5 страницы). Структура связного текста:

1. **Эмпирический факт.** `no-pruning` — единственная компонента с отрицательным Δ для всех 7 моделей (mean -22,0 п.п., max -31,2 п.п. на Claude-3.5). Самый крупный и самый стабильный эффект во всём кубе.

2. **Объяснение через AST-инвариант.** Pruner отбраковывает тесты с нарушениями инвариантов (тип, импорт, сигнатура, неинициализированные переменные) до `go test`. Без него один невалидный тест ломает компиляцию пакета и аннулирует уже работавшие тесты других файлов внутри запуска — кросс-функциональный негативный эффект.

3. **Усиление через анализ типов отказов.** В 1008 запусках основного куба категории «компилируется-но-падает» (`all-failed`) и «частично проходит» (`partial`) **отсутствуют полностью**. Все отказы концентрируются в категории «нет валидных тестов». Из этого следуют два важных вывода:
   - Pruner функционально корректен: не пропускает тесты, которые упали бы на runtime
   - Главная причина отказа — отсутствие сгенерированных тестов вовсе, а не плохие assertions

4. **Связь с заявленной новизной.** Это сильный аргумент в пользу pruner-модуля как оригинального вклада работы (Глава 3.6.3): количественно показано, что без него агент теряет 22 п.п. успешности в среднем, и отсутствие partial-failure состояний подтверждает, что фильтр не пропускает «токсичные» тесты.

### §5.3.3. Не-монотонное поведение остальных компонент

Связный текст с встроенными примерами:

- Компоненты `no-types`, `no-smart-diff`, `no-structured-feedback` показывают умеренное среднее влияние (mean Δ от -1,8 до -5,7 п.п.) с **разнознаковым разбросом** между моделями
- Конкретные примеры: `no-structured-feedback` ухудшает Llama (-12,5 п.п.) и Qwen-7B (-4,2), но улучшает DeepSeek (+6,2) и Gemini (+2,1); `no-types` критичен для DeepSeek (-16,7 п.п.) и Qwen3-30B (-10,4), но почти не влияет на GPT-4o-mini (+2,1)
- **Главная гипотеза подраздела.** Утилитарность дополнительного контекста зависит от того, умеет ли модель этим контекстом пользоваться. Передовая Gemini в среднем не нуждается в дополнительном фидбеке (пишет валидный тест с первой попытки); средний тир систематически выигрывает; floor-модель не имеет capability чтобы воспользоваться даже хорошим контекстом
- **Научное наблюдение.** Полезность вспомогательных компонент пайплайна **обратно коррелирует с capability базовой модели** — это нетривиальный эмпирический результат, согласующийся с (но не следующий из) литературой по prompt engineering

### §5.3.4. Цикл расширения покрытия как trade-off

`no-coverage` — единственная конфигурация с **положительным** Δ на всех 7 моделях (mean +12,8 п.п., max +29,2 на Llama). Это **не означает бесполезности компонента**, а отражает trade-off:

- Cover-loop оптимизирует **branch coverage**, а не file_macro (детально в §5.5.2)
- Cover-loop **стабилизирует результат** между запусками: `no-coverage` имеет σ = 18,4 п.п. (max среди конфигураций), `full` — 13,3 п.п. (детально в §5.5.7)
- Цена: вторая итерация cover-loop иногда ломает уже работающий файл (новый тест нарушает компиляцию или имя совпадает с существующим)
- Соответственно cover-loop работает как **двойной модуль**: оптимизация ветвлений + стабилизация результата ценой умеренного снижения file_macro
- Для CI-сценария, где предсказуемость критична, такой trade-off оправдан

### §5.4.1. Постановка вопроса и proxy-baseline

Способен ли разработанный агент-харнесс компенсировать слабость более простых LLM? «Чистый» bare-LLM baseline (модель без обвязки) не входит в основной куб — это будущая работа. Однако конфигурация `no-pruning` представляет собой **минимально-функциональный pipeline** (LLM + `go test`, без AST-фильтрации, ближе всего к «голому LLM с обвязкой компиляции» из имеющихся точек данных) и используется как **proxy-baseline**. Подраздел заканчивается **caveat'ом**: у proxy всё ещё есть retry-цикл, structured feedback и типы — истинный baseline отложен в будущую работу. Несмотря на это наблюдаемые эффекты настолько систематические, что общее заключение остаётся обоснованным.

### §5.4.2. Эффект уравнивания: от proxy к full

Главная подглава §5.4 (~2 страницы), включает:

- **Таблица T6** — full vs no-pruning по run_success (7 строк: model, no-pruning %, full %, Δ п.п., кратное улучшение). Среднее +43,5 п.п. по всем; +37,5 п.п. в среднем тире (без пола и frontier)
- **График G3** — paired bars no-pruning vs full per model с подписанными Δ. Самый драматичный график главы; упоминается одной фразой в тексте
- **Таблица T7** — переход категорий (floor <25%, mid 25–75%, prod-ready 75–95%, frontier ≥95%). Показывает: 5 из 7 моделей меняют категорию благодаря harness; особенно драматичны Claude (floor → prod-ready через всю промежуточную) и Gemini (mid → frontier)

Связный текст опровергает две очевидные гипотезы и подтверждает третью:
1. **«Полное закрытие гэпа» — НЕТ.** Спред Gemini–Qwen-7B расширяется с 62,5 до 75 п.п. (frontier имеет запас, floor — нет)
2. **«Слабые выигрывают абсолютно больше» — НЕТ.** Самые большие приросты у Claude (+62,5 п.п.) и mid-tier GPT-4o-mini / Qwen3-Coder-30B (+50). Floor-модель получает только +25 п.п. — harness не может компенсировать отсутствие способности писать синтаксически валидный Go-код
3. **«Уравнивание моделей среднего тира в production-ready категорию» — ДА.** Это и есть ключевой результат RQ3

### §5.4.3. Diminishing returns и практическая рекомендация

- **Diminishing returns at the ceiling.** Кратное улучшение (full / no-pruning) убывает с ростом исходной capability: Claude 6×, Qwen3-30B 5×, GPT-4o-mini 3×, Llama 2,8×, DeepSeek 2,3×, Gemini 1,6×. Frontier-модель уже близка к functional ceiling — дополнительный прирост наименьший среди всех
- **Практическая рекомендация для CI:** для модели среднего тира выбор определяется уже стоимостью и латентностью; различия в capability существенно нивелируются агентом. Это снижает чувствительность пайплайна к выбору LLM-провайдера и к ценовой/доступностной волатильности
