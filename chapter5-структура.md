## §5.1. Постановка эксперимента (~3-4 страницы)

### §5.1.1. Цели и гипотезы

Перечисляются исследовательские вопросы (RQ), извлекаемые из задач во введении:
- **RQ1**: Какова зависимость качества автоматически сгенерированных тестов от выбора языковой модели?
- **RQ2**: Каков вклад каждого из шести компонент пайплайна (smart-diff, types-context, structured-feedback, pruner, coverage-loop, naturalness) в итоговое качество?
- **RQ3**: Способен ли агент-харнесс компенсировать слабость более простых LLM, приводя их к качеству, сопоставимому с frontier-моделями?
- **RQ4**: Какова экономическая эффективность функционального кэширования в типичных CI/CD-сценариях?
- **RQ5**: Какие методологические ограничения cross-provider бенчмаркинга через OpenAI-compatible API необходимо учитывать в будущих сравнительных исследованиях?

### §5.1.3. Конфигурации ablation-исследования

6 конфигураций, эквивалентных в смысле reproducibility, но различающихся одним отключённым компонентом:

| ID | Описание | Что отключено |
|----|----------|---------------|
| `full` | Полный пайплайн | — |
| `no-types` | Без типизированного анализа | `internal/typed`: интерфейсы, граф вызовов |
| `no-smart-diff` | Без семантического фильтра diff | `internal/diff`: пропускаются комментарии, форматирование |
| `no-structured-feedback` | Без машиночитаемой обратной связи | `internal/testjson`: ошибки передаются как plain-text |
| `no-pruning` | Без AST-фильтрации невалидных тестов | `internal/pruner` |
| `no-coverage` | Без coverage-loop расширения | `internal/coverage` итерации не выполняются |

Конфигурация `no-cache` отсутствует в основной матрице ablation сознательно (см. §5.5: тождественна `full` при `git clean -fdx` между запусками; вынесена в отдельный микро-эксперимент).

### §5.1.5. Параметры воспроизводимости

- `temperature = 0` (детерминированный sampling, насколько это возможно для LLM)
- `seed = 42 + run_index` (1, 2, 3) для разных runs одной (model, repo, config)
- `provider.only` фиксирован для каждой модели (исключение auto-routing)
- `provider.allow_fallbacks = false` (failure лучше чем подмена провайдера)
- `git reset --hard <head> && git clean -fdx` между runs (исключение state leakage)
- WSL2/Ubuntu как execution environment (исключение Windows-specific тестовых ошибок)

### §5.1.6. Метрики

(Краткое перечисление, детали в §3.6, на которые ссылаемся.)
- **Success rate** — доля runs, в которых сгенерирован ≥ 1 валидный (компилирующийся, проходящий) тест
- **Tests validated** — общее число валидных тестов
- **Branch coverage** — `go cover` per affected functions
- **Diff coverage** — кастомная метрика покрытия только изменённых строк
- **Mutation score** — собственный модуль, мутации в изменённом коде
- **Naturalness** — 5 sub-метрик: assertion ratio, no-assertion%, duplicate-assertion%, test-name score, var-name score
- **Token efficiency** — prompt+completion токенов на один валидный тест
- **Agent wall-clock** — время от запуска агента до завершения (per (repo, config))

### §5.1.7. Самотестирование инструмента (один абзац!)

> «Программная реализация SAGA сопровождается собственным набором юнит-тестов. На момент проведения экспериментальной оценки в репозитории~\cite{saga-repo} 36~файлов автотестов охватывают 23~внутренних пакета; среднее покрытие по~операторам составляет~72,1\%, при~этом критичные модули (\texttt{pruner}, \texttt{cache}, \texttt{branchcov}, \texttt{testjson}) имеют покрытие выше~90\%. Это~служит дополнительным свидетельством корректности тех~компонент, чей~вклад количественно оценивается в~настоящей главе.»

### §5.1.8. Воспроизводимость и публикация артефактов

## §5.3. Ablation-исследование: вклад компонент (~5-6 страниц)

### Содержание

**Таблица T4**: per-model × per-config success rates (7 × 6 матрица).

```
                         full   no-types  no-smart  no-feedback  no-prune  no-cov
Qwen 7B (floor)          25%       17%       25%           8%        0%     17%
Qwen 30B                 62%       46%       67%          58%       12%     54%
Llama 70B                71%       54%       67%          54%       25%     71%
DeepSeek V3              58%       46%       71%          71%       25%     62%
GPT-4o-mini              75%       79%       83%          83%       25%     62%
Claude 3.5 Haiku         75%       62%       75%          79%       12%     71%
Gemini 3 Flash          100%       96%      100%         100%       62%     96%
```

**График G2**: heatmap (зелёный — выше full, красный — ниже full).

### §5.3.1. Универсальная критичность pruner-а (центральная находка)

- При удалении pruner success rate падает на **40-71 п.п.** на всех моделях
- Это самый универсальный и крупный эффект во всём ablation-кубе
- Объяснение через AST-инвариант: LLM может писать частично-некорректные тесты, и без AST-фильтрации даже один невалидный тест ломает всю компиляцию пакета
- Связь с §5.4: именно pruner — главный механизм practical equalizer effect

### §5.3.2. Не-монотонное поведение остальных компонент

- `no-structured-feedback` иногда **улучшает** результат на сильных моделях (GPT-4o-mini, Claude 3.5)
- `no-smart-diff` ведёт себя аналогично
- Объяснение через capability tier: чем сильнее модель, тем меньше она нуждается в обвязке; для слабых моделей обвязка критична
- **Это самостоятельный научный finding**, согласующийся с TestPilot (Schäfer 2024) и ChatTester (Yang 2024)

### §5.3.3. Умеренный, но стабильный вклад типизированного анализа

- `no-types` стабильно ухудшает результат на 10-20 п.п. на mid-tier моделях
- На frontier (Gemini) почти не влияет: модель сама умеет инферировать типы
- На floor (Qwen 7B) — небольшое ухудшение (с 25 до 17): floor модель не умеет использовать types-context эффективно даже если он есть

### §5.3.4. Coverage-loop: цена неоднозначна

- `no-coverage` практически не влияет на success rate
- Однако влияет на branch/diff coverage (нужны отдельные данные)
- Цена: coverage-loop может удваивать prompt-токены за счёт второго прохода

## §5.4. Practical equalizer effect (~4-5 страниц, **главный contribution**)

### Содержание

**Таблица T5**: сравнение `full` vs `no-pruning` (proxy bare-LLM) по 7 моделям.

| Модель | no-prune | full | Δ п.п. | × ratio | tier change |
|--------|---------:|-----:|-------:|--------:|-------------|
| Qwen 7B | 0% | 25% | +25 | ∞ | useless → usable |
| Qwen 30B | 12% | 62% | +50 | 5.0× | marginal → strong |
| Llama 70B | 25% | 71% | +46 | 2.8× | usable → strong |
| DeepSeek V3 | 25% | 58% | +33 | 2.3× | usable → strong |
| GPT-4o-mini | 25% | 75% | +50 | 3.0× | usable → strong |
| Claude 3.5 | 12% | 75% | +63 | 6.0× | marginal → strong |
| Gemini | 62% | 100% | +38 | 1.6× | strong → excellent |

**График G3**: paired bars (no-pruning vs full per model), с подписанным spread'ом.

### Три формулировки гипотезы (центральная часть)

1. **«Closes the gap полностью»** — НЕТ. Спред расширяется с 62.5 до 75 п.п.
2. **«Слабые выигрывают абсолютно больше»** — НЕТ. Mid-tier (Claude 3.5, Qwen-30B, GPT-4o-mini) выигрывают +50-63 п.п., floor только +25.
3. **«Уравнивает mid-tier в production-ready категорию»** — ✅ ДА. 5 моделей сходятся в 58-75% strong-tier.

### Diminishing returns at the ceiling

- Frontier-модель Gemini в bare-LLM-режиме (62.5%) уже близка к functional ceiling
- Дополнительный прирост 1.6× (наименьший среди всех моделей)
- Согласуется с ASTER §6 и TestPilot §5 для аналогичных доменов

### Caveat (важно для §5.7)

- `no-pruning` — это **proxy**, а не bare-LLM
- Истинный bare-LLM baseline («одно сообщение LLM, без агента») — future work
- Direction-of-effect устойчив: bare-LLM был бы ниже no-pruning, значит реальные Δ еще больше

### Практическая рекомендация (отсылка к §6 conclusion)

> «Для CI-сценария выбор LLM в mid-tier диапазоне определяется уже стоимостью и латентностью; capability-различия нивелируются агентом. Это снижает чувствительность пайплайна к ценовой/доступностной волатильности рынка моделей и устраняет vendor lock-in.»

## §5.5. Микро-эксперимент: эффективность кэширования (~1.5-2 страницы)

### Содержание

**Таблица T6**: cold / warm-same-head / warm-shifted-head на `gpt-4o-mini × gorilla/mux`.

| Run | head SHA | prompt | completion | duration | cache hits/misses |
|-----|----------|-------:|-----------:|---------:|------------------|
| A (cold) | f30e... | 100% | 100% | 100% | 0/2 |
| B (warm same head) | f30e... | 69% | 57% | 48% | 2/0 |
| C (warm shifted) | db9d... | 95% | 110% | 92% | 0/3 |

### Ключевые числа

- Warm-same-head: **−52% wall-clock**, **−31% prompt tokens**, **−43% completion tokens**
- 100% cache-hit rate (2/2 функции из predicted)
- Корректная инвалидация на shifted head (новый коммит → 0 hits, 3 misses)

### Практическое следствие

- В CI-сценарии «PR пересобирается без изменений» (immutable-branches) экономит до 50% wall-clock
- Не включён в основную ablation matrix: при `git clean -fdx` между конфигами `no-cache ≡ full`, измерять там нечего

## §5.7. Угрозы валидности (~3-4 страницы)

### §5.7.1. Configuration trap (max_tokens × cross-provider routing)

- Эмпирически обнаружена нетривиальная угроза валидности cross-provider бенчмарков через OpenAI-compatible API
- Дефолт `max_tokens=4096` соблюдается строго одними провайдерами (Anthropic native), игнорируется другими (Bedrock, Google AI Studio)
- Это привело к систематическому занижению результатов одной из моделей (`claude-haiku-4.5`), которая была исключена из основной выборки
- **Самостоятельный методологический результат настоящей работы**
- Рекомендация для будущих исследований: либо `max_tokens` ≥ 16k, либо adaptive cap

### §5.7.2. Отсутствие bare-LLM baseline (future work)

- В качестве proxy для bare-LLM использована конфигурация `no-pruning`
- Истинный baseline (LLM single-shot без обвязки) не выполнен по бюджетным соображениям
- Direction-of-effect устойчив: реальные Δ от агента ≥ репортируемых

### §5.7.3. Платформенная специфика

- Все эксперименты выполнены в WSL2/Ubuntu
- На Windows native некоторые тестовые pipelines падают по PATH/symlink-причинам, не связанным с агентом
- Заявленные результаты воспроизводимы на Linux/macOS

### §5.7.4. Юридический статус OpenRouter в РФ

- На момент проведения эксперимента (май 2026) OpenRouter применяет географические ограничения для российских IP-адресов
- Все эксперименты выполнялись через VPN (Clash Verge TUN-mode)
- Это **не блокирует** ни воспроизведение, ни легальное использование инструмента в других юрисдикциях
- Альтернативные провайдеры (DeepInfra, Together AI, Novita прямо) не ограничивают РФ-IP

### §5.7.5. Размер выборки и статистическая значимость

- 3 runs на каждую (model, repo, config) — позволяет оценить только грубое варьирование
- Для сильных claims (например, +50 п.п. от pruner) этого достаточно
- Для тонких сравнений (например, ranking двух близких моделей) статистическая значимость ограничена

## Выводы по главе 5 (~1-2 страницы)

Структурированное перечисление **6 центральных contributions**:

1. **Capability ladder**: эмпирическое расслоение 7 LLM шести вендоров от 15% до 92% success rate, обеспечивающее scaling-анализ
2. **Universal pruner criticality**: главный движок системы; -40 до -71 п.п. при удалении на всех моделях
3. **Non-monotonic ablation behavior**: компоненты помогают слабым моделям, но мешают (или нейтральны) сильным; зависимость утилитарности от capability
4. **Practical equalizer effect**: функциональное уравнивание mid-tier моделей в единую "production-ready" категорию (58-75% success); устраняет vendor lock-in для CI
5. **Cache effectiveness**: −52% wall-clock на warm-same-head; корректная инвалидация на shifted head
6. **Configuration trap finding**: методологическая угроза валидности cross-provider бенчмаркинга через OpenAI-compatible API

И **2 практические рекомендации**:
- Pruner — единственный неотключаемый компонент при production-применении
- Выбор LLM в mid-tier диапазоне определяется ценой/латентностью, не capability
