# Кандидатные репозитории для экспериментального датасета

Шорт-лист по 8 слотам из §6.1 плана. На каждый слот — 2–3 проекта, чтобы
было из чего выбрать после клонирования и проверки сборки. Конкретные
SHA пар `base..head` фиксируются на следующем шаге, после клонирования
финалистов и просмотра их recent merged PR.

Условные обозначения:

- **LOC** — приблизительный размер продового кода (без `_test.go` и
  vendor) на момент составления списка. Цифры округлены до тысяч.
- **CI** — есть ли в репо рабочий GitHub Actions / Travis с `go test`.
- **Сложность сборки** — насколько вероятно, что `go test ./...`
  пройдёт без дополнительной настройки.

Все выбранные проекты соответствуют базовым техническим критериям:
Go ≥ 1.21, `go.mod` в корне, активная разработка (последний коммит < 6
месяцев), MIT/BSD/Apache лицензия.

## Критерии отбора PR на каждом репо

После клонирования финалистов на каждом проекте просматриваем 20–30
последних merged PR и ищем те, что соответствуют:

**Хочется (positive):**

- Изменение **1–4 файлов в продовом коде** (`internal/`, `pkg/`, корень).
- Один из изменённых файлов — содержит публичный API или активную
  бизнес-логику (не просто `String()`-метод или константа).
- В PR были изменения и в самих `_test.go` (значит это разработческий PR
  с реальной логикой, а не косметика).
- Размер изменений в продовых файлах: 20–200 строк (`+` минус `−`).
- PR прошёл CI на момент merge.

**Не подходит (negative):**

- Только обновление зависимостей (`go.mod`, `go.sum`).
- Только косметика: переименования, форматирование, комментарии.
- Чисто сгенерированный код (`*.pb.go`, `mocks/`, `swagger/`).
- Чисто документация (`README.md`, `docs/`).
- Изменения в `vendor/`.
- Чисто тестовые PR (изменения только в `_test.go`).

**Особое внимание:**

- Чтобы `head` собирался: `git checkout <head> && go build ./... && go test ./...`
  должен пройти зелёным. Проверяем перед фиксацией `head` в датасет.
- Чтобы был достаточно «недавним»: PR не старше 12 месяцев на момент
  старта эксперимента (важно из-за совместимости с актуальными версиями
  Go и отсутствия deprecated API).

## Действия на следующем шаге

1. Клонировать 8 рекомендуемых проектов в `workdir/`.
2. На каждом — `gh pr list --state merged --limit 30` (или GitHub web).
3. По критериям выше выбрать 1 финальный PR на репо + 2 запасных.
4. Зафиксировать `base` и `head` SHA в `dataset.yaml`.
5. Прогнать `go build && go test ./...` на `head` каждого выбранного
   PR — если падает, переключиться на запасной.
6. Закоммитить итоговый `dataset.yaml` в `testgen-agent-experiments`.
