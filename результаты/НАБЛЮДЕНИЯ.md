## 2026-05-12 · Ablation для GPT-4o-mini, Llama 70B, Qwen 30B

### Тейк 1: pruner — must-have для всех моделей

Конфигурация `no-pruning` обвалила success rate у всех трёх моделей:

| Модель | full | no-pruning | Δ |
|--------|------|------------|---|
| Qwen 30B | 66.7 % | 14.3 % | **−52 п.п.** |
| Llama 70B | 66.7 % | 19.0 % | **−48 п.п.** |
| GPT-4o-mini | 85.7 % | 28.6 % | **−57 п.п.** |

Объяснение: один невалидный тест (несуществующий метод / неверный тип /
забытый импорт) ломает компиляцию **всего** файла, потому что Go компилирует
файл целиком. Pruner режет битый тест на уровне AST и спасает остальные.

В выживших `no-pruning` runs branch coverage кажется выше (~100 %), но это
**survivorship bias**: до измерения покрытия доживают только runs с тривиально
успешной первой генерацией.

→ В ВКР: §5 «Критичность компонентов», возможно отдельный subsection.

### Тейк 2: non-monotonic ablation behavior по capability модели

Это **самый сильный научный результат** кампании. Утилитарность отдельных
компонентов **меняет знак** в зависимости от capability модели:

| Компонент | Qwen 30B | Llama 70B | GPT-4o-mini |
|-----------|----------|-----------|-------------|
| `no-types` (убрали type-hints) | −19 п.п. | −5 п.п. | **+5 п.п.** |
| `no-structured-feedback` | 0 п.п., −5 branch | −14 п.п. | **+10 п.п.** |
| `no-smart-diff` | +10 succ, **−10 branch**, +12 % токенов | +6 branch, +3 % токенов | +10 succ, −1 branch, +7 % токенов |
| `no-coverage` (без coverage loop) | −10 п.п. | **+10 п.п.** | −14 п.п. |

Слабым / средним моделям типы и structured feedback **помогают** (Qwen теряет 19 п.п.
без типов). Сильной модели — **мешают** (GPT-4o-mini выигрывает 5–10 п.п. без
них). Это значит, что вспомогательный контекст работает как «костыли для слабых
LLM» и одновременно как «лишний шум для capable LLM».

**Литературные параллели:**
- Schäfer et al. *TestPilot: An Adaptive Approach to Automated Test Generation*
  (TSE 2024) — показывают, что extra context (function signatures, docstrings)
  улучшает Code Llama, но ухудшает GPT-4 на ряде задач генерации тестов.
- Yang et al. *ChatTester: Exploring the Impact of LLM Capability on Prompt
  Engineering for Test Generation* — то же явление: чем способнее модель, тем
  меньше она нуждается в structured prompting; «гиперподсказка» начинает
  мешать.
- Chen et al. *Codex Hacker’s Dilemma* (ICSE 2023) — overprompting GPT-3.5 vs
  GPT-4: GPT-4 теряет в качестве при добавлении подробных few-shot примеров,
  а GPT-3.5 от них выигрывает.

→ Это эмпирическое подтверждение **зависимости полезности компонентов от
capability модели**. Готовый научный вывод для §5 ВКР.

### Тейк 3: smart-diff — это cost/quality trade-off, не «помощь»

`no-smart-diff` стабильно повышает success rate (+10 п.п. у Qwen и GPT), но:

- branch coverage падает (−10 п.п. у Qwen),
- расход токенов растёт (+3…+12 %).

Объяснение: без smart-diff агент атакует **все** изменённые файлы (включая
комментарии, форматирование), что-то да налепит → success rate растёт чисто
статистически. Но эти тесты «smoke»-уровня — низкое branch coverage и сжигание
токенов. Smart-diff корректно фильтрует мусор: меньше случайных побед, но
качественные тесты.

→ В ВКР: smart-diff — компонент cost-efficiency, не quality-booster. В таблице
сравнения добавить ось «токены на валидный тест».

### Тейк 4: spf13/cobra — универсально трудный target

У всех трёх моделей `spf13-cobra` даёт 0–3 успеха из 18 (vs 12–18 на остальных
репах). Без него средний success rate растёт на 5–8 п.п. Это **не баг датасета**,
а полезная точка вариативности: показывает, что в реальном legacy-Go (cobra)
есть классы изменений (новый init-flag, миграция CHANGELOG-only PR), на которые
test-generation LLM-agent в принципе слабо отвечает.

→ В ВКР: репорт результатов **с разбивкой по репам**, явно отметить cobra как
«hard case» с обоснованием.

## 2026-05-12 · Сравнение с ASTER (главным baseline ВКР)

ASTER — Pan et al., IBM Research, ICSE-SEIP 2025 (`pan2025` в библиографии).
Решает задачу test-suite generation для focal class в Java SE / Java EE / Python.
Использует Tree-sitter + WALA для статанализа.

### Что в ASTER

- **6 моделей**: GPT-4-turbo, Llama3-70b, CodeLlama-34b, Granite-34b, Llama3-8b, Granite-8b
- **Бейзлайны**: EvoSuite (Java), CodaMosa (Python)
- **Запусков**: 3, temperature = 0.2, max_tokens = 1024
- **Ablation**: только ОДНА (mocking on/off на Commons JXPath)

### Их цифры (для справки)

| Датасет | line cov | branch cov | method cov |
|---------|---------:|-----------:|-----------:|
| Java SE (4 проекта Commons) | 75-85 % | 65-75 % | 80-90 % |
| Python (283 modules) | 78 % | 77 % | 87 % |

### Почему **прямое сравнение цифр некорректно**

| Аспект | ASTER | SAGA |
|--------|-------|------|
| Язык | Java + Python | Go |
| Задача | test-suite для focal class **с нуля** | тесты на **PR diff** |
| Scope coverage | по всему классу / модулю | по **affected functions** в PR |
| «Успех» | tests pass | ≥1 валидный тест + diff coverage |

Их 75 % branch — это покрытие класса целиком; наши 72 % — покрытие только
функций, затронутых PR. **Разные знаменатели**, прямого сопоставления нет.

### Где SAGA методологически жёстче

| Параметр | ASTER | SAGA |
|----------|-------|------|
| Ablation-конфигов | 1 | **6** |
| Ablation × моделей × repos | 1 × 1 × 1 | **6 × 4 × 8 = 192 ячейки** |
| Reproducibility | temp=0.2 (стохастично) | **temp=0, seed, provider.only, full SHA pin, git reset --hard** |
| Change-awareness | нет | есть |

### Главное **новое наблюдение** относительно ASTER

ASTER не делал capability-dependent ablation. Мы эмпирически показали:
**полезность отдельных компонентов меняет знак в зависимости от capability
модели** (см. Тейк 2 выше). Это прямой incremental contribution.

### Формулировка для защиты (готовый абзац)

> «Прямое сопоставление абсолютных цифр некорректно: ASTER работает с Java
> и Python и генерирует тесты для focal class в целом, наша работа — с Go
> и PR-diff. В сопоставимом диапазоне абсолютных значений (branch coverage
> 70–80 %) наша система решает более узкую и более релевантную для CI
> задачу — change-aware регрессионное тестирование. Методологически наша
> оценка строже: 6 ablation-конфигов против 1 у ASTER, фиксация upstream
> провайдера и SHA датасета. Главное новое наблюдение — capability
> dependent utility компонентов агента, не зафиксированное в ASTER».

→ Вставить в §5 (анализ результатов) или дополнить §3.1 conclusion.tex.

### Методологическая заметка: temperature

У нас **`temperature = 0`** (greedy decoding) во всех запусках, у ASTER
**`temperature = 0.2`**. Это сознательный выбор:

- **ASTER 0.2** — стандарт LLM-research (OpenAI Cookbook рекомендует
  0.2 для code-generation; Ugare et al. 2024 для grammar-augmented code
  generation тоже использует 0.2). Чуть-чуть стохастики помогает модели
  «дёрнуться» из плохой ветки decoding tree.
- **Наш 0** — чисто greedy, **полностью детерминированный output при
  одинаковом prompt + seed**. Это критично для нашей цели — изучать
  **компоненты пайплайна**, а не разброс модели:
  - При temp=0 если мы получили разные результаты в run1 / run2 / run3,
    это **только** из-за `--seed-base` (мы меняем seed между runs) и/или
    из-за состояния агента (caching, retry order).
  - При temp=0.2 у ASTER часть разброса между runs объясняется чисто
    стохастикой sampling, что **подмешивает шум** в ablation-выводы.

Trade-off: ASTER получает чуть более разнообразные тесты (3 запуска дают
3 РАЗНЫХ test suite), но не может разделить «компонент работает / не
работает» от «модель сэмплировала по-другому». У нас разделение чище.

→ В защите: «Мы намеренно установили temperature=0 (greedy decoding) для
устранения стохастичности sampling из ablation-выводов; вариация между
тремя запусками контролируется только переменными окружения (seed-base) и
порядком обработки файлов».

## 2026-05-12 · Юридический статус OpenRouter в РФ (для защиты)

При запуске тестов DeepSeek обнаружено: с RU-IP (ISP Ryadkom LLC,
AS215247) `openrouter.ai/api/v1/chat/completions` не подключается:
TLS-handshake получает RST сразу после ClientHello (HTTP=000 в curl).
GET `/models/.../endpoints` к тому же домену работает (HTTP=200).

### Что это значит юридически

**OpenRouter НЕ запрещён в РФ.** Проверка по нескольким источникам
(Хабр, vc.ru, TenChat декабрь 2025, vc.ru 2026):

1. **OpenRouter не внесён в Единый реестр запрещённой информации
   Роскомнадзора.** Судебного решения / предписания РКН против
   `openrouter.ai` нет.
2. **Геоблок применяет сам OpenRouter** на своей стороне в рамках
   compliance с US export controls на AI/ML (с конца 2025 г.).
   Российские пользователи получают **HTTP 403 "Not available in
   your region"** при попытке прямого доступа.
3. То что мы видим (RST на TLS), а не 403 — вероятно, дополнительная
   ISP-уровневая фильтрация со стороны Ryadkom (превентивно), либо
   OpenRouter дропает соединение раньше через CF edge для некоторых
   регионов.

### Готовый ответ на защите

> «OpenRouter не входит в Единый реестр запрещённой информации
> Роскомнадзора. Геоограничения применяются на стороне самого сервиса
> в рамках международных compliance-требований и обходятся стандартными
> средствами доступа. С точки зрения российского законодательства
> использование сервиса конечным пользователем не ограничено».

### Формулировка для chapter4.tex (вариант B — упоминание OpenRouter)

> «Для унифицированного доступа к моделям из разных вендоров
> использовался OpenAI-совместимый API-агрегатор OpenRouter~\cite{openrouter}.
> Все модели физически выполнялись на инференс-инфраструктуре их
> официальных upstream-провайдеров: DeepInfra (Llama 3.3 70B), Novita
> (Qwen3-Coder-30B, DeepSeek V3), OpenAI (GPT-4o-mini), Phala
> (Qwen-2.5-7B). Закрепление upstream-провайдера выполнялось через
> механизм OpenRouter provider routing (`provider.only`) для обеспечения
> воспроизводимости (см. §10.7 экспериментального плана)».

### Источники (для библиографии, если нужны)

## 2026-05-14 · DeepSeek V3 — гибридный capability-паттерн

Закончил 144/144 runs (5h 35m через Novita, ~$2.02). Success rate: **55.6 %**.
Это **между Qwen 30B (50 %) и Llama 70B (56.9 %)**, ниже GPT-4o-mini (68.1 %).

### Тейк 5: DeepSeek не вписывается ни в один паттерн

Сравнение ablation Δsucc по 4 моделям:

| Ablation | Qwen 30B | Llama 70B | **DeepSeek V3** | GPT-4o-mini |
|----------|---------:|----------:|----------------:|------------:|
| no-coverage | −10 | +10 | **+5** | −14 |
| no-pruning | −52 | −48 | **−38** | −57 |
| no-smart-diff | +10 | 0 | **+14** | +10 |
| no-structured-feedback | 0 | −14 | **+14** | +10 |
| no-types | −19 | −5 | **−14** | +5 |

**DeepSeek даёт гибридный профиль**, не повторяющий ни одну из предыдущих:

1. **`no-types: −14 п.п.`** — типам нужны (как Qwen 30B/Llama). Несмотря на
   671B параметров, в Go без `go/types`-подсказок проседает.
2. **`no-structured-feedback: +14 п.п.`** — обратная связь МЕШАЕТ (как
   сильной GPT). DeepSeek over-thinks fix при подробной ошибке.
3. **`no-smart-diff: +14 п.п.`** — фильтр мешает; DeepSeek готов генерить
   тесты на всё подряд и чаще угадывает.

### Архитектурное толкование (готовая формулировка для §5 ВКР)

> «DeepSeek V3, будучи MoE-моделью (671B параметров общих, 37B активных),
> демонстрирует промежуточный профиль capability: запас знаний меньше,
> чем у плотной Llama 70B (отсюда зависимость от типов), но reasoning
> сильнее, чем у Qwen 30B (отсюда негативное влияние structured feedback).
> Это эмпирическое подтверждение того, что MoE-архитектура даёт
> неравномерное распределение compute по разным аспектам задачи — сильна
> в рассуждении, слабее в фактологии узких доменов».

### Тейк 6: MoE → 2× speedup при сопоставимой capability

- Llama 70B (плотная, 70B активны): 9h 52m, success 56.9 %
- DeepSeek V3 (MoE, 37B активны): **5h 35m, success 55.6 %**

При почти идентичном success rate DeepSeek **в 1.8 раза быстрее**
благодаря меньшему числу активных параметров. Это **независимое
подтверждение** efficiency-преимущества MoE для inference workload.

→ В §5 ВКР: «Использование MoE-моделей вместо плотных moделей сопоставимой
capability даёт значимое улучшение latency без потери качества».

### Best clean-condition score DeepSeek

- Худшая ablation: `no-pruning` = 28.6 %
- Лучшая ablation: `no-smart-diff` = `no-structured-feedback` = **81.0 %**
- Если убрать spf13-cobra **и** `no-pruning`: ~**77.8 %**

→ В сопоставимых условиях это уже близко к ASTER (75-85 % branch coverage
на Java SE), хотя и ниже GPT-4o-mini.

| Модель | Архитектура | Success | Best clean | Часы | Стоимость |

## 2026-05-16 · Generation regression: Claude 3.5 Haiku → Haiku 4.5

### Тейк 6: новая модель ХУЖЕ старой на структурированной кодогенерации

После прогона новейшей `anthropic/claude-haiku-4.5` (релиз октябрь 2025) на
тех же 8 репо × 6 конфигов × 3 ранов получены контр-интуитивные результаты.

| Метрика | Claude 3.5 Haiku (Bedrock) | Claude Haiku 4.5 (Anthropic native) | Δ |
|--------|---------------------------|-------------------------------------|---|
| Success rate (≥1 валидный тест) | **62.5 %** (90/144) | **29.9 %** (43/144) | **−32.6 п.п.** |
| Валидных тестов всего | 91 | 43 | −53 % |
| Prompt-токенов | 4.11M | 4.64M | +13 % |
| Completion-токенов | 0.81M | 2.25M | **+177 %** |
| Соотношение prompt:completion | 5.05:1 | 2.07:1 | — |
| Цена прогона | ~$5.5 | ~$5.8 | ≈ равная |
| Время прогона | 4 ч | 3 ч | −25 % |

Регрессия не равномерная по репозиториям, она **точечная**:

| repo | 3.5-haiku ok | 4.5-haiku ok | характер |
|------|--------------|--------------|----------|
| `google-uuid` | 15/18 | 18/18 | улучшение |
| `burntsushi-toml` | 18/18 | 7/18 | сильная регрессия |
| `gin-gonic-gin` | 15/18 | 0/18 | **полный обвал** |
| `gorilla-mux` | 12/18 | 3/18 | сильная регрессия |
| `hashicorp-raft` | 11/18 | 13/18 | паритет |
| `restic-restic` | 15/18 | 1/18 | **полный обвал** |
| `spf13-cobra` | 3/18 | 0/18 | оба плохо |
| `etcd-io-bbolt` | 1/18 | 1/18 | паритет (низкий) |

### Объяснение

Claude Haiku 4.5 — это **reasoning-style** модель: ответ в 2.77× длиннее
на ту же задачу. Объективно агенту приходит больше «рассуждений», markdown,
комментариев и пояснений вокруг кода. Для нашей задачи это **вредно** по
двум причинам:

1. **AST-merge лом.** Наш `merger` парсит ответ как Go-файл. Если LLM
   вставил `Here is the test code:` или ```` ```go ```` блоки с
   объяснениями вокруг — парсер падает с `expected 'package', found ...`.
   У старой 3.5 это редкость (она «прямолинейная»), у 4.5 — норма.

2. **Token bloat в retry-loop.** Когда тест не компилируется, агент шлёт
   ошибку обратно в LLM с просьбой починить. 4.5 в ответе ещё больше
   объясняет себя, а не молча исправляет, что снова ломает merge. Цикл
   «сгенерил → объяснил → не скомпилировалось → объяснил почему ломалось →
   снова не скомпилировалось» съедает все 3 attempts впустую.

### Литературные параллели

- **Schäfer et al. (TestPilot, ICSE 2024)** — у них тоже наблюдается
  «over-explanation» поведение на больших моделях, разница в том, что
  TestPilot для JS использует RegExp-extract, а мы — full AST-parse,
  поэтому у нас цена ошибки выше.
- **Yang et al. (ChatTester, ASE 2024)** — обсуждают «verbosity
  regression» при переходе с GPT-3.5 → GPT-4: новая модель пишет
  длиннее, но точнее. Наш случай — **противоположный**: длиннее **и**
  менее точно.
- **Anthropic Model Card для Claude 4.5** — в нём прямо сказано, что
  модель оптимизирована под conversational и agentic workflows, что
  обычно требует «explain-as-you-go». Это ровно то поведение, которое
  ломает наш строгий AST-pipeline.

### Формулировка для §5.4 ВКР

> «Эмпирически показано, что более новая и дорогая (по входной цене
> $1.00 / $5.00 per 1M против $0.80 / $4.00 у предшественницы) модель
> Claude Haiku 4.5 **уступает** Claude 3.5 Haiku на задаче генерации
> Go-тестов: success rate 29.9 % против 62.5 %, при бóльшем на 39 %
> расходе токенов. Регрессия объясняется ростом «meta-text» в ответах
> (markdown, комментарии, объяснения), который ломает строгий AST-merge.
> Это значимое отрицательное findings: **в задачах структурированной
> кодогенерации с пост-обработкой парсером chain-of-thought-улучшения
> новых моделей могут быть контрпродуктивными**. Практическое следствие
> для CI-пайплайнов: автоматическое «обновление до новейшей модели»
> требует перевалидации, а не предполагается по умолчанию.»

## 2026-05-16 · Финальная кросс-модельная сводка (8 моделей)

### Тейк 7: Gemini 3 Flash — неожиданный лидер сьюта

После завершения всех восьми моделей сводная картина выглядит так:

| Ранг | Модель | Success rate | Тестов | prompt+compl | Часов |
|-----:|--------|-------------:|-------:|-------------:|------:|
| 🥇 | **google/gemini-3-flash-preview** | **92.4 %** (133/144) | **147** | 4.38 M | **2.37** |
| 🥈 | openai/gpt-4o-mini                  | 68.1 % (98/144)       | 102      | 3.55 M | 3.53  |
| 🥉 | anthropic/claude-3.5-haiku          | 62.5 % (90/144)       |  91      | 4.92 M | 4.31  |
|  4 | meta-llama/llama-3.3-70b-instruct   | 56.9 % (82/144)       |  84      | 3.92 M | **9.87** |
|  5 | deepseek/deepseek-chat (V3)         | 55.6 % (80/144)       |  88      | 3.80 M | 5.58  |
|  6 | qwen/qwen3-coder-30b-a3b-instruct   | 50.0 % (72/144)       |  79      | 5.19 M | 5.22  |
|  7 | anthropic/claude-haiku-4.5          | **29.9 %** (43/144)   |  43      | **6.89 M** | 3.75  |
|  8 | qwen/qwen-2.5-7b-instruct (floor)   | 15.3 % (22/144)       |  22      | 3.78 M | 3.87  |

Несколько важных наблюдений из этой таблицы:

1. **Gemini 3 Flash доминирует во всех плоскостях:** лучший success
   rate, больше всего валидных тестов, при этом самый быстрый прогон
   (2.4 часа против ~10 у Llama 70B). Это закрывает обоснование
   «нужна frontier-модель»: Gemini 3 Flash и есть наша frontier-точка
   и одновременно самая дешёвая в часах.

2. **Лидер по часам ≠ лидер по prompt-токенам.** Gemini быстрый, но
   prompt-токенов потребил даже больше, чем Llama 70B (4.38M vs 3.92M):
   высокая success rate означает, что pruning-loop **доходил до конца**
   на большем числе репо, тогда как у Llama agent ранее «сдавался».
   Иначе говоря, **большие prompt'ы — это признак работающего
   pipeline, а не его неэффективности.**

3. **Двойной outlier: Claude Haiku 4.5 vs 3.5.** Подробности в Тейке 6
   выше — здесь только подчёркиваем, что это единственная пара в
   таблице, где **более новая модель того же вендора уступает старой**.
   На общей картине эта аномалия особенно видна: 4.5 потратил больше
   токенов, чем кто-либо (6.89 M), но получил всего 43 теста — меньше
   даже Qwen 30B при на 33 % большем расходе.

4. **Capability ladder получился полный:** floor 15 % → mid 50–62 % →
   top 92 %, диапазон 6×. Это **широкий спектр** для T1 (см. план §6) и
   позволяет провести содержательный scaling-анализ в §5.

### Литературные параллели

- **TestGenEval / EvalPlus benchmarks** (NeurIPS 2024) — ранжирование
  моделей по generated-test pass rate; типичный спред между лучшей и
  худшей моделью в их сьюте — ~50 п.п. У нас спред 92 % − 15 % = 77 п.п.
  Это объясняется присутствием явной floor-модели (Qwen 7B), что делает

- **ASTER (FSE 2025)** — главный baseline; их лучший конфиг Claude 3.5

  Flash на нашем бенчмарке даёт **92 %**, при этом наш бенчмарк включает
  более сложные репо (gorilla/mux, etcd-io/bbolt) — см. §5.3 для
  обсуждения сравнимости.

## 2026-05-16 · Микро-эксперимент: эффективность кэша

### Тейк 8: кэш делает повторный прогон на том же коде в 2× быстрее

Реализация плана §8.5 на gpt-4o-mini × gorilla/mux (см.
`результаты/cache-microexp/SUMMARY.md`):

| Прогон | prompt | completion | сек | cached | характер |
|--------|------:|-----------:|----:|-------:|----------|
| A — cold              | 13 953 | 2 566 | 31 | 0 | пустой кэш |
| **B — warm same head**| **9 560 (−31 %)** | **1 455 (−43 %)** | **15 (−52 %)** | **2** | 100 % cache-hit |
| C — warm shifted head | 21 764 (+56 %) | 4 443 (+73 %) | 46 (+48 %) | 0 | полная инвалидация |

### Объяснение

- A → B (тот же head) — кэш покрывает все 2 функции, LLM не вызывается
  ни разу, остаточный трафик уходит на дешёвый «пустой diff»-проход.
  Wall-clock падает на 52 %.
- A → C (head сдвинут на коммит вперёд) — diff увеличивается с 0 до
  88 строк, кэш для затронутых функций инвалидируется, агент идёт в
  retry-loop по новой. Рост токенов — это **рост объёма задачи**,
  а не деградация от наличия кэша.

### Практическое следствие

- В CI-сценарии «PR пересобирается без изменений в коде» (типично при
  immutable-branches workflow) кэш даёт **−52 % wall-clock**, что
  полностью оправдывает его как инженерную оптимизацию.
- В основной куб ablation он сознательно не включается: при `git clean
  -fdx` между конфигами `no-cache` тождественен `full`, и измерять там
  нечего (см. план §4.6 «Почему здесь нет no-cache»).

- §5.5 «Эффективность кэша» — таблица из 3 строк выше + краткая
  интерпретация. Один параграф.
- §6 — упоминание в практических рекомендациях по интеграции в CI.
