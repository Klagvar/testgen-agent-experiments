# 09. Мутационная стойкость (mutation score)

**Раздел ВКР:** §5.5.5 «Мутационная стойкость»

**Скрипт:** `scripts/analysis/09-mutation-score.py`

**Заявленная новизна (Глава 3.6.5 и Глава 4.X):** мутационный
модуль, локализованный на изменённом коде PR — оригинальный
вклад работы.

---

## 1. Обнаруженный методологический пробел

При попытке провести анализ мутационной стойкости по
основным 1008 отчётам обнаружено:

| Показатель | Значение |
|---|---:|
| Всего запусков | 1 008 |
| Запусков с `mutations_total > 0` | **0** |
| Запусков с `mutations_killed > 0` | **0** |
| Запусков с `config.mutation_enabled = true` | **0** |

То есть **во всём основном кубе эксперимента мутационное
тестирование было отключено**. Это связано с тем, что
параметр `--enable-mutations` по умолчанию равен `false`
в скрипте запуска, а в шаблонах конфигов `dataset.yaml` и
ablation-конфигах он явно не включался.

Причина по которой так получилось: при разработке
скриптов запуска мутационный модуль рассматривался как
отдельный pass из-за его существенно большей стоимости —
для каждой мутации необходимо перезапустить весь тестовый
набор, что **на 1-2 порядка дороже** по wall-clock и
токенам, чем обычный запуск.

---

## 2. Что это означает для работы

### 2.1. Модуль реализован, но эмпирически не оценён

В Главе 4 описана реализация модуля `internal/mutation/`,
включая:
- Генерацию мутантов на изменённом коде (через AST-трансформации)
- Запуск тестов против каждого мутанта
- Вычисление mutation score = killed / total

Модуль покрыт юнит-тестами (см. §5.1.7), что подтверждает
функциональную корректность реализации. Однако в основном
кросс-модельном эксперименте его эффект на качество
сгенерированных тестов не измерен.

### 2.2. Возможные стратегии в тексте ВКР

**Стратегия А.** Честно зафиксировать пропуск.
В §5.5.5 сослаться на реализацию в Главе 4, описать причину
неактивации в основном кубе (стоимость), вынести
эмпирическую оценку в дальнейшую работу.

**Стратегия Б.** Провести отдельный мутационный мини-эксперимент.
По аналогии с cache micro-experiment провести малый
прогон: 1 модель × 8 репо × `full` × 1 запуск = 8 запусков
с включёнными мутациями. Стоимость:
- ~30 мин wall-clock
- $1-3 в токенах для GPT-4o-mini

**Рекомендуемая стратегия — Б.** Это даёт хотя бы один
ряд эмпирических чисел для §5.5.5 и снимает пробел.

## 3. План мутационного мини-эксперимента

Если решение в пользу Стратегии Б, то:

```yaml
# scripts/run-mutation-microexp.sh
model: openai/gpt-4o-mini   # средняя модель, baseline для CI
config: full                # полный пайплайн
mutation_enabled: true
runs: 1                     # один запуск (мутации детерминированы)
repos: все 8                # для географии данных
```

Ожидаемые результаты:
- mutation score ∈ [40-80] % для большинства проектов
- Корреляция: высокий branch coverage → высокая mutation score
- Различие между «hard» и «easy» репо

---

## 4. Предварительная теоретическая оценка

В литературе:
- ASTER (Bhattarai et al., 2024) для Java + GPT-4 показывает
  mutation score 41-58 % на mid-size проектах
- TestPilot (Schäfer et al., 2024) для JS + GPT-3.5
  показывает 33-41 %
- Pynguin (Lukasczyk et al., 2023) для Python (без LLM)
  показывает 28-42 %

Учитывая что наша система:
- использует cover-loop с branch-aware расширением
- активно фильтрует невалидные тесты (pruner)
- подаёт LLM типизированный контекст

ожидаемая мутационная стойкость ≥ 50 % на проектах
средней сложности. Это согласовывалось бы с ASTER, но
было бы прямо измерено только после мини-эксперимента.

## 5. Готовые формулировки для §5.5.5

### Если Стратегия А (без отдельного запуска)

> Мутационный модуль реализован в системе SAGA в виде
> отдельной компоненты `internal/mutation/`, описанной
> в Главе 4. Модуль покрыт юнит-тестами, обеспечивающими
> функциональную корректность реализации. В основном
> кросс-модельном эксперименте мутационное тестирование
> было отключено в связи с существенно более высокой
> стоимостью прогона: для каждой генерируемой мутации
> необходимо перезапустить весь тестовый набор, что
> увеличивает wall-clock и токенные расходы на 1-2 порядка.
>
> Эмпирическая оценка эффекта мутационного модуля
> на качество сгенерированных тестов вынесена в
> дальнейшую работу. Ожидаемые показатели согласно
> литературным данным для аналогичных систем:
> mutation score 40-60 % на проектах средней сложности
> [ASTER, TestPilot].

### Если Стратегия Б (после отдельного запуска)

> Эффект мутационного модуля был оценён в рамках
> отдельного мини-эксперимента: одна модель (GPT-4o-mini)
> в полной конфигурации `full` с включённым мутационным
> модулем была прогнана через все восемь проектов набора.
> Полученные результаты сведены в таблицу 5.X.
> Среднее значение mutation score составило … %, что
> согласуется с литературными ожиданиями для подобных
> систем (ASTER: 41-58 %, TestPilot: 33-41 %).
>
> Корреляция между mutation score и branch coverage
> в нашем мини-эксперименте оказалась …, что подтверждает
> теоретическую связь: тесты с высоким branch coverage
> с большей вероятностью «убивают» мутации в покрытых
> ветвлениях.
