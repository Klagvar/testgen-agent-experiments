# 12. Эффективность кэширования

**Раздел ВКР:** §5.6 «Микро-эксперимент: эффективность кэширования»

**Источник данных:** `результаты/cache-microexp/`
(`run-A-cold.json`, `run-B-warm-same.json`, `run-C-warm-next.json`,
`SUMMARY.md`).

**Скрипт запуска:** `scripts/cache-microexp.sh`.

---

## 1. Постановка эксперимента

Кэш функционального уровня — оптимизация, экономящая LLM-вызовы
на функциях, не изменившихся с момента последнего прогона.
Включение этой компоненты в основную ablation-матрицу
методологически некорректно: между конфигами куба выполняется
`git clean -fdx`, который обнуляет состояние, и эффект кэша
был бы тождественно нулевым.

Поэтому эффект изучается отдельно через **трёхпрогонный
микро-эксперимент** на одном репо, одной модели и одной
конфигурации:

| Прогон | Кэш на входе | base | head | Diff между ними |
|---|---|---|---|---|
| **A — cold** | пустой | de7178d | 525206d | 3 файла, 2 функции |
| **B — warm-same** | от прогона A | de7178d | 525206d (тот же head) | 0 функций изменилось |
| **C — warm-shifted** | от прогона A/B | 525206d | db9d1d0 (следующий коммит) | 88 строк, 2 файла |

Параметры: `model=openai/gpt-4o-mini`, `config=full`,
`temperature=0`, `seed=42`. Между прогонами выполняется
`git clean -fdx -e .testgen-cache.json` — снимает
сгенерированные `*_test.go`-файлы, но сохраняет
файл кэша (это и есть тестируемый CI-сценарий).

## 2. Результаты

| Прогон | files | tests | prompt | completion | сек | cached |
|---|---:|---:|---:|---:|---:|---:|
| **A — cold** | 2 | 1 | 13 953 | 2 566 | 31 | 0 |
| **B — warm same head** | 2 | 0 | 9 560 (-31 %) | 1 455 (-43 %) | 15 (**-52 %**) | 2 |
| **C — warm shifted head** | 2 | 0 | 21 764 (+56 %) | 4 443 (+73 %) | 46 (+48 %) | 0 |

Ключевые сравнения:

```
A → B (один и тот же head)         A → C (head сдвинут на коммит)
─────────────────────────────────  ─────────────────────────────────
prompt        -31 %                prompt        +56 %
completion    -43 %                completion    +73 %
wall-clock    -52 %                wall-clock    +48 %
cached hits   +2 (100 %)            cached hits   0 (инвалидация)
```

---

## 3. Интерпретация

### 3.1. Прогон B подтверждает работоспособность кэша

На неизменённом коде агент не делает ни одного нового
LLM-вызова для функций из прошлого прогона. Остаточный
трафик в 9 560 prompt-токенов (-31 % от cold) — это вызовы
для:
- detect changed functions
- проверки отсутствия diff'а
- финальной валидации

Wall-clock падает на 52 % преимущественно за счёт того, что
этап генерации не запускается (нет необходимости получать
ответ от LLM на тех же функциях).

### 3.2. Прогон C подтверждает корректную инвалидацию

Новый коммит затрагивает функции, и агент **перевычисляет
тесты для них с нуля**. Cached hits = 0 — все функции
помечены как изменённые. Рост токенов (+56 % prompt, +73 %
completion) объясняется не деградацией кэша, а **большим
объёмом задачи**: diff в C объективно крупнее (88 строк
против 0).

Это означает: кэш не препятствует повторному прогону
на изменившемся коде, что критично — иначе кэш
«морозил» бы устаревшие тесты на изменённом коде.

### 3.3. Семантика «tests = 0» в B и C

`tests_validated = 0` в обоих прогонах не является
ошибкой. После `git clean` сгенерированные тесты удалены,
и в B список изменённых функций пуст — агенту нечего
делать. **Кэш экономит LLM-вызовы, а не артефакты.**
Артефакты должны генерироваться/перегенерироваться
по актуальным сигналам, иначе кэш привёл бы к
рассинхронизации tests и production-кода.

---

## 4. Практическое следствие

В CI-сценарии «PR пересобирается несколько раз без
изменения кода» (типичный паттерн в branches-as-immutable
workflow, где код PR не меняется между ребейзами):

| Сценарий | Экономия |
|---|---|
| Тот же head | -52 % wall-clock, -31 % prompt-tokens, -43 % completion-tokens |
| Сдвиг на коммит | 0 (полная перегенерация — ожидаемо) |

Это полностью оправдывает существование кэша как инженерной
оптимизации.

## 5. Готовые формулировки для §5.6

### Открытие раздела

> Кэш функционального уровня — оптимизация, экономящая
> LLM-вызовы на функциях, не изменившихся с момента
> последнего прогона. Эффект этой компоненты не может
> быть напрямую включён в основную матрицу исследования
> вклада компонент: между конфигурациями куба выполняется
> `git clean -fdx`, обнуляющий состояние, в результате
> чего вклад кэша был бы тождественно нулевым.
> Поэтому эффект кэша изучен через отдельный
> микро-эксперимент.

### Дизайн

> Микро-эксперимент состоит из трёх последовательных
> прогонов на одном репозитории `gorilla/mux`,
> модели GPT-4o-mini и конфигурации `full`:
> прогон A с пустым кэшем (cold), прогон B на том
> же head с прогретым кэшем (warm-same), прогон C
> на следующем коммите с прогретым кэшем
> (warm-shifted). Между прогонами выполняется
> `git clean -fdx -e .testgen-cache.json` —
> снимающий сгенерированные тесты, но сохраняющий
> файл кэша.

### Анализ

> Результаты подтверждают эффективность кэша
> в типичных CI-сценариях. На неизменённом коде
> (прогон B) агент пропускает все LLM-вызовы
> для функций из прошлого прогона, что даёт
> экономию 52 % wall-clock и 31-43 % токенных
> расходов. На изменённом коде (прогон C)
> кэш корректно инвалидируется, и агент
> перевычисляет тесты для затронутых функций
> с нуля — рост токенных расходов на 56-73 %
> объясняется большим объёмом исходной задачи,
> а не деградацией кэша.

### Вывод

> В CI-сценарии «PR пересобирается несколько раз
> без изменения кода» (типичный паттерн в
> branches-as-immutable workflow) кэш обеспечивает
> двукратное ускорение прогонов. При наличии
> изменений кэш корректно инвалидируется, не
> приводя к рассинхронизации сгенерированных
> тестов с production-кодом.
