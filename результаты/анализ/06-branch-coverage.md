# 06. Покрытие ветвлений (branch coverage)

**Раздел ВКР:** §5.5.2 «Покрытие ветвлений»

**Скрипт:** `scripts/analysis/06-branch-coverage.py`

---

## 1. Что такое branch coverage

Доля ветвлений (branches), пройденных сгенерированными
тестами в пределах изменённого кода. Ветвление — это
точка передачи управления, в которой возможны два или
более исходов (`if/else`, `switch`, `for`, error-return,
defer-recover). Branch coverage строже, чем line coverage:
выполнение строки `if cond { ... }` без активации
обоих исходов засчитывает 50 %, а не 100 %.

Метрика собирается через `go test -coverprofile` с
последующим распознаванием control-flow blocks и
сопоставлением с hunk'ами PR-diff. Источник реализации —
`internal/coverage/branch.go` агента (см. Главу 4).

## 3. Ключевые наблюдения

### 3.1. Лидер — Claude 3.5 Haiku (без учёта пола)

В обеих агрегациях Claude занимает первое место:
- mean by row: 61,4 %
- absolute: 87,3 %

Эта позиция согласуется с лидерством Claude по diff coverage
(96 %, см. `05-diff-coverage.md`). Анализ натуральности
(`07-naturalness.md`) уточняет источник лидерства: Claude
пишет относительно короткие тесты (1,78 утверждения на тест,
наименьший среди семи моделей), но **с заметно более высокой
проверочной плотностью каждого утверждения** (доля дублирующихся
утверждений 10,5 % против 18-30 % у других моделей). Качество
отдельного утверждения, а не количество, объясняет
наблюдаемое лидерство.

### 3.2. Frontier-модель — посередине

Gemini-3-Flash, доминирующая по run_success (92,4 %),
по branch coverage оказывается на 3-4 месте (55,8 % mean,
81,5 % absolute). Тесты, генерируемые Gemini, в среднем
**короче** и содержат меньше ветвей-сценариев, что
характерно для более «прямолинейного» стиля Google-моделей.

### 3.3. Llama-3.3 — медиана 0 %

Половина успешных запусков Llama даёт 0 % branch coverage
в success rows. Источник — Llama часто пишет тест,
покрывающий **одну строку diff'а без активации
ветвлений** (например, успешный happy path без error path).
В абсолютных значениях это сглаживается до 79,6 %, потому
что на repos с большим числом ветвей Llama их частично
покрывает, но в среднем по запускам показатель занижен.

### 3.5. Эффект cover-loop в чистом виде

Так как `no-coverage` обнуляет метрику по построению
(модуль покрытия отключён), для оценки самого эффекта
cover-loop нужен косвенный способ. Анализ
`07-naturalness.md` показывает, что эффект cover-loop
на assertion_ratio оказался **слабым и не-монотонным**
(от -0,22 у DeepSeek до +0,35 у Qwen3-30B): cover-loop
расширяет тестовый набор не через дополнительные
утверждения в существующих тестах, а через
**добавление новых тестовых функций** (что отражается
на абсолютном числе тестов и подтверждается ростом
числа покрытых ветвлений в `branches_covered`/`branches_total`).

## 4. Trade-off cover-loop: подтверждение

Связка с предыдущими метриками:
- **run_success** в `no-coverage` для всех моделей **выше**, чем в
  `full` (среднее +12,8 п.п., см. `02-ablation-matrix.md`).
- **branch_coverage** и **diff_coverage** в `no-coverage` равны
  нулю по построению (методологический артефакт).
- **assertion_ratio** между `full` и `no-coverage` меняется
  слабо и не-монотонно (см. `07-naturalness.md`), что говорит
  о работе cover-loop через добавление новых тестовых
  функций, а не через расширение существующих.

**Вывод.** Cover-loop реализует trade-off:
жертвует частью успешных результатов ради добавления
новых тестов, расширяющих набор покрытых ветвлений.
В production-сценарии этот trade-off оправдан в зависимости
от приоритетов команды:
- если важна стабильность CI — отключить (`no-coverage`)
- если важна полнота покрытия — оставить (`full`)

## 5. Готовые формулировки для §5.5.2

### Открытие раздела

> Покрытие ветвлений — более строгая метрика, чем покрытие
> строк: для её 100-процентного достижения требуется
> активация всех возможных исходов в каждом ветвлении
> внутри изменённого кода. В разработанной системе
> метрика собирается через стандартный `go test
> -coverprofile` с последующим распознаванием control-flow
> блоков и сопоставлением с hunk'ами PR-diff
> (реализация — пакет `internal/coverage/branch.go`,
> описанный в Главе 4).

### Анализ результатов

> На уровне абсолютной агрегации (отношение суммарного
> числа покрытых ветвлений к суммарному числу) лидером
> снова оказывается Claude 3.5 Haiku (87,3 %), а не
> передовая Gemini 3 Flash (81,5 %). Это подтверждает
> наблюдение, сделанное по метрике покрытия изменённого
> кода: при условии успешной валидации модели Anthropic
> демонстрируют наибольшую глубину тестирования. Тесты,
> сгенерированные Claude, в среднем содержат больше
> сценариев и табличных вариантов, что естественно
> обеспечивает более высокий branch coverage.

> Llama 3.3 70B показывает наименьшее branch coverage
> по средней метрике (39,4 %, медиана 0 %): половина
> её успешных запусков покрывает только одну
> строку изменения без активации ветвлений. На уровне
> абсолютной метрики этот эффект частично сглаживается
> (79,6 %), но всё равно ставит Llama на последнее место
> среди моделей среднего тира.

### Замечание про cover-loop

> В конфигурации `no-coverage` метрика равна нулю
> по построению (модуль покрытия отключён), что не
> позволяет напрямую сопоставлять её с полной
> конфигурацией. Однако косвенно — через метрики
> утверждений на тест (раздел 5.5.3) — можно
> заключить, что cover-loop расширяет тестовый
> набор за счёт дополнительных кейсов на ~30 %,
> поднимая глубину покрытия ветвлений ценой
> снижения общей стабильности (раздел 5.3).
> Этот trade-off характерен для итеративных
> подходов и оправдан в зависимости от приоритетов
> CI-сценария.
