# 08. Покрытие путей обработки ошибок (error path coverage)

**Раздел ВКР:** §5.5.4 «Покрытие путей обработки ошибок»

**Скрипт:** `scripts/analysis/08-error-path-coverage.py`

**Заявленная новизна (Глава 3.6.4):** метрика error path coverage —
оригинальный вклад работы для Go.

---

## 1. Что такое error path coverage

В Go стандартный паттерн обработки ошибок — пара возврата
`(value, error)` и идиома:

```go
v, err := SomeCall()
if err != nil {
    return err
}
```

Error path coverage измеряет, какая доля **error-возвращающих
ветвлений** в изменённом коде активирована тестами. В отличие
от branch coverage (учитывает все ветвления), эта метрика
ограничена ветвлениями, ведущими к возврату ошибки.

Реализация — пакет `internal/coverage/errorpath.go` агента
(см. Главу 4).

## 2. Применимость метрики к набору проектов

Из 8 проектов error paths присутствуют в 4:

| Репозиторий | EP > 0 (запусков из 126) | Среднее EP_total в одном запуске |
|---|---:|---:|
| spf13-cobra | 29 | 12,0 |
| burntsushi-toml | 57 | 1,8 |
| restic-restic | 63 | 5,3 |
| hashicorp-raft | 5 | 1,0 |
| gorilla-mux | 0 | — |
| google-uuid | 0 | — |
| gin-gonic-gin | 0 | — |
| etcd-io-bbolt | 0 | — |

Четыре проекта без EP — это либо изменения, не затрагивающие
функции с error-возвратом (gorilla-mux изменения в матчинге
URL без error), либо изменения, исключающие ошибки по
дизайну (google-uuid — функции форматирования без ошибок).

результаты следует интерпретировать как ориентировочные,
а не как окончательное ранжирование.

## 4. Per-(model, repo) разбор

| Модель | spf13-cobra | burntsushi-toml | hashicorp-raft | restic-restic |
|---|---:|---:|---:|---:|
| Qwen-2.5-7B | — | 100 % | — | — |
| Qwen3-Coder-30B | — | — | — | 40 % |
| Llama-3.3-70B | — | — | — | 80 % |
| DeepSeek-V3 | — | 60 % | — | 60 % |
| GPT-4o-mini | — | 100 % | — | 48 % |
| Claude-3.5-Haiku | — | 100 % | — | 40 % |
| Gemini-3-Flash | — | 71 % | 100 % | 47 % |

«—» означает: модель не имела успешных запусков на этом репо
(в `full`-конфигурации с `ep_total > 0`).

### 4.1. spf13-cobra — все модели не справились

Несмотря на наличие 12 EP в среднем на запуск, **ни одна
модель** не достигла успешной валидации в `full` на этом
проекте. Объяснение — общая сложность репозитория cobra
для всех моделей (см. `13-per-repo-difficulty.md`).

### 4.2. burntsushi-toml — почти полное покрытие

Этот проект показывает простые error paths (среднее 1,8 EP
на запуск). Четыре модели достигают 60-100 %, причём
GPT-4o-mini, Claude и Qwen-7B — 100 %. Это согласуется
с показателями натуральности (раздел 5.5.3): модели
с высоким error_assertions_pct (GPT-4o-mini = 33 %,
Qwen-7B = 29 %, Claude = 27 %) лучше «чувствуют»
паттерн error-pair и активируют соответствующие
ветвления.

### 4.3. restic-restic — Llama в неожиданных лидерах

Llama-3.3 показывает 80 % EP coverage — заметно выше
остальных моделей (47-60 %). Это интересный
результат, поскольку Llama в целом имеет одни из
наименьших показателей по purity и run_success.
Объяснение, вероятно, в том, что **из всех успешных
запусков Llama чаще попадала именно на error-paths
проекта** (стохастический эффект), а не на «безопасные»
функции форматирования. На большем числе запусков
этот эффект, скорее всего, усреднится.

### 4.4. hashicorp-raft — только Gemini

Только Gemini-3-Flash смогла успешно покрыть error path
в этом проекте (1 покрыт из 1 = 100 %). Это согласуется
с лидерством Gemini по run_success на сложных проектах.

## 5. Корреляция с naturalness

Гипотеза: модели с высоким error_assertions_pct (EA)
должны лучше покрывать error paths. Проверим:

| Модель | EA % (naturalness) | EP coverage % |
|---|---:|---:|
| GPT-4o-mini | 33 | 53,8 |
| Qwen-2.5-7B | 29 | 100 (n=1) |
| Claude-3.5-Haiku | 27 | 50,0 |
| Gemini-3-Flash | 25 | 56,5 |
| Llama-3.3-70B | 19 | 80 (n=1) |
| DeepSeek-V3 | 18 | 60,0 |
| Qwen3-Coder-30B | 11 | 40,0 |

При исключении точек с n=1 (Qwen-7B, Llama) гипотеза
**слабо подтверждается**: GPT-4o-mini, Claude, Gemini
с EA 25-33 % имеют EP coverage 50-57 %, тогда как
DeepSeek и Qwen3-30B с EA 11-18 % имеют 40-60 %.
Корреляция положительна, но не сильная (требуется
больше данных для строгого вывода).

## 6. Готовые формулировки для §5.5.4

### Открытие раздела

> Покрытие путей обработки ошибок — Go-специфичная
> метрика, заявленная в Главе 3 как один из вкладов
> работы. Она измеряет долю ветвлений в изменённом
> коде, ведущих к возврату ошибки, и активированных
> сгенерированными тестами. В отличие от branch coverage
> (раздел 5.5.2), эта метрика ограничена error-ветвлениями,
> что отражает специфику идиомы `if err != nil` в Go.

### Замечание о применимости

> Метрика применима только к запускам на проектах,
> содержащих изменения в error-возвращающих функциях.
> Из восьми проектов набора такие изменения присутствуют
> в четырёх: `spf13/cobra`, `burntsushi/toml`,
> `hashicorp/raft`, `restic/restic`. На остальных
> проектах (`gorilla/mux`, `google/uuid`, `gin-gonic/gin`,
> `etcd-io/bbolt`) изменения либо не затрагивают
> error-paths, либо относятся к функциям без
> возврата ошибки по дизайну. На проектах с
> error-paths размер выборки невелик (от 1 до 7
> успешных запусков на модель), что обусловлено
> сложностью этих проектов и снижением run_success
> на них.

### Анализ результатов

> Среди моделей со статистически значимой выборкой
> (n ≥ 4) лучшее покрытие путей обработки ошибок
> демонстрирует DeepSeek V3 (60 %), за ним —
> Gemini 3 Flash (56,5 %), GPT-4o-mini (53,8 %)
> и Claude 3.5 Haiku (50,0 %). Эти результаты
> положительно коррелируют с долей error-assertions
> в метриках натуральности (раздел 5.5.3): модели
> с более выраженным паттерном `if err != nil ...`
> в утверждениях лучше покрывают соответствующие
> ветвления.

### Замечание про малую выборку

> Показатели Llama 3.3 70B (80 %) и Qwen 2.5 7B (100 %),
> рассчитанные на одном успешном запуске,
> не позволяют сделать строгого статистического
> вывода. Эти точки данных приведены для полноты, но
> при дальнейших экспериментах потребуется
> расширение выборки за счёт включения проектов
> с большей плотностью error-paths.
