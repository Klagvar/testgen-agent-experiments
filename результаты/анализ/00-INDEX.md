# Анализ результатов эксперимента — навигация

Все данные **пересчитаны с нуля** по 1008 машиночитаемым отчётам
(7 моделей × 8 проектов × 6 конфигураций × 3 запуска), хранящимся в
`результаты/raw/`.

Каждый файл этого каталога — самостоятельная единица анализа,
содержит:
1. Краткое определение метрики
2. Точный скрипт пересчёта (или ссылку на скрипт в `scripts/`)
3. Таблицы с числами
4. Интерпретацию
5. Готовые формулировки для соответствующего раздела ВКР

## Список файлов

### Часть 1. Сводные результаты по моделям

- **`01-capability-ladder.md`** — главная таблица 7 моделей
  по доле успешных запусков (для §5.2)

### Часть 2. Исследование вклада компонент

- **`02-ablation-matrix.md`** — полная матрица 7 × 6 (для §5.3 и G2)
- **`03-pruner-criticality.md`** — детальный разбор отключения
  модуля удаления невалидных тестов (для §5.3.3)
- **`04-equalizer-effect.md`** — анализ
  «полный пайплайн против минимально-функционального» (для §5.4 и G3)

### Часть 3. Дополнительные метрики качества

- **`05-diff-coverage.md`** — покрытие изменённого кода (§5.5.1)
- **`06-branch-coverage.md`** — покрытие ветвлений (§5.5.2)
- **`07-naturalness.md`** — 5 подметрик натуральности (§5.5.3)
- **`08-error-path-coverage.md`** — покрытие путей обработки
  ошибок (§5.5.4)
- **`09-mutation-score.md`** — мутационная стойкость, включает результаты
  отдельного целевого прогона 7 моделей × 8 репо × `full` × 1 run (§5.5.5,
  данные в `результаты/mutation-microexp/`)
- **`10-token-efficiency.md`** — экономика по токенам
  и фронт Парето стоимость–качество (§5.5.6)
- **`11-variance-stability.md`** — стабильность результатов
  между независимыми запусками (§5.5.7)

### Часть 4. Целевые прогоны и дополнительный анализ

- **`12-cache-microexp.md`** — эффективность кэширования (§5.6)
- **`13-per-repo-difficulty.md`** — сложность репозиториев,
  объяснение различий (для §5.1.2)
- **`14-failure-modes.md`** — анализ типов ошибок при неуспешных
  запусках (бонус, может войти в §5.3 или §5.5)

## Принципы

1. Все числа в этих файлах — **single source of truth** для текста
   ВКР. При расхождении приоритет у этих файлов.
2. Файлы не зависят друг от друга, можно читать в произвольном
   порядке.
3. Скрипты лежат в `scripts/analysis/<нумерация>-<название>.py`.
   Каждый принимает стандартное расположение `результаты/raw/`
   и выводит результат в stdout.
4. Терминология русифицирована: «исследование вклада компонент»
   вместо «ablation study», «модуль удаления невалидных тестов»
   вместо «pruner» в основном тексте (англ. термин в скобках при
   первом упоминании).
5. Цвета и художественное оформление в таблицах не используются.
