# Micro-experiment: эффективность кэша

**Модель:** `openai/gpt-4o-mini` &nbsp;·&nbsp; **репо:** `gorilla/mux` &nbsp;·&nbsp;
**ablation:** `full` &nbsp;·&nbsp; **temperature:** 0 &nbsp;·&nbsp; **дата:** 2026-05-16

Дизайн соответствует плану §8.5: измерить экономию LLM-токенов, времени
и долю переиспользованной кэш-записей при разных сценариях повторного
прогона агента на одном и том же репозитории.

## Дизайн

| Прогон | Кэш на входе | base | head | Diff |
|--------|--------------|------|------|------|
| **A — cold** | пустой | `de7178d` | `525206d` | 3 файла, 2 функции |
| **B — warm same head** | от прогона A | `de7178d` | `525206d` (та же пара) | 0 функций изменилось |
| **C — warm shifted head** | от прогона A/B | `525206d` | `db9d1d0` (следующий коммит) | 88 изменённых строк, 2 файла |

Между прогонами выполняется `git clean -fdx -e .testgen-cache.json` —
это снимает сгенерированные `*_test.go`-файлы, но **сохраняет** файл
кэша (без сохранения кэш не имел бы смысла на следующей итерации
CI-цикла, что и есть тестируемый сценарий).

## Результаты

| Прогон | files | tests | prompt | completion | сек | cached записей |
|--------|------:|------:|-------:|-----------:|----:|---------------:|
| **A — cold**            | 2 | 1 | **13 953** | **2 566** | **31** | 0 |
| **B — warm same head**  | 2 | 0 |  9 560 (−31 %) |  1 455 (−43 %) | 15 (−52 %) | 2 |
| **C — warm shifted head** | 2 | 0 | 21 764 (+56 %) | 4 443 (+73 %) | 46 (+48 %) | 0 |

Сравнение метрик:

```
A → B (один и тот же head)         A → C (head сдвинут на коммит вперёд)
─────────────────────────────────  ───────────────────────────────────────
prompt        −31 %                prompt        +56 %
completion    −43 %                completion    +73 %
wall-clock    −52 %                wall-clock    +48 %
cached hits   +2 (100 % покрытие)  cached hits   0   (полная инвалидация)
```

`tests = 0` в B и C — это не ошибка: после `git clean` сгенерированные
файлы удалены, и поскольку в B список «изменённых функций» пуст
(diff = 0), агент не имеет работы — он не «достаёт» прежние тесты из
кэша, а только не делает LLM-вызов для уже виденных функций. Файл с
тестами при этом пропадает, потому что для него нет нового источника
истины. **Это сознательная семантика: кэш экономит LLM-вызовы, а не
артефакты — артефакты должны генерироваться/перегенерироваться по
актуальным сигналам.**

## Интерпретация

1. **B (warm same head) подтверждает работоспособность кэша.**
   На неизменённом коде агент не делает ни одного нового LLM-вызова
   для функций из прошлого прогона; экономия prompt-токенов 31 %
   объясняется только маленьким остаточным трафиком на выявление
   diff'а и проверку отсутствия изменений. Wall-clock падает на 52 %
   преимущественно за счёт того, что mutation-testing и pruning не
   запускаются в полном объёме.

2. **C (warm shifted head) показывает корректную инвалидацию кэша.**
   Новый коммит затрагивает функции, и агент перевычисляет тесты для
   них с нуля. Рост токенов (+56 % prompt, +73 % completion) объясняется
   тем, что diff в C объективно крупнее, чем в A (88 изменённых строк
   против 0), а pruning-loop у `gpt-4o-mini` на крупных diff'ах
   делает больше итераций «исправь ошибку компиляции». То есть это не
   деградация от кэша, а **больший объём задачи**.

3. **Практическое следствие.** Для CI-сценария «PR пересобирается
   несколько раз без изменения кода» (что является типичным паттерном
   в branches-as-immutable workflow) кэш даёт −52 % wall-clock. Это
   полностью оправдывает его существование как инженерной оптимизации,
   при том что в основной куб ablation он сознательно не включён (см.
   §8.5 плана) — иначе с `git clean -fdx` между конфигами он был бы
   тождественно нулевым.

## Источники

- **JSON-отчёты прогонов:** `run-A-cold.json`, `run-B-warm-same.json`,
  `run-C-warm-next.json` (в этой же папке).
- **Полный лог:** `логи/cache-microexp.log` (в корне репо
  эксперимента).
- **Скрипты:**
  - `scripts/cache-microexp.sh` — последовательно выполняет A → B → C
    с сохранением `.testgen-cache.json` между прогонами;
  - `scripts/cache-microexp-C.sh` — дозапуск только C, если A и B уже
    есть (использовался при отладке после исправления полного SHA для
    `db9d1d0`).
